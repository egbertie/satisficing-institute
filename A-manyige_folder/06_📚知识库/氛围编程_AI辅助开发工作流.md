# 氛围编程：AI辅助开发工作流指南

> 来源：https://mp.weixin.qq.com/s/Z4supEjCIQc85NsaYrKDLA  
> 作者：张诗瑶、吴越（清华大学沈阳教授团队）  
> 收录时间：2026-03-15  
> 适用场景：非技术人员用AI进行软件开发

---

## 核心观点

氛围编程不是随意地让AI写代码。**规范化的工作流**意味着：在每个阶段用最适合的工具，给AI最准确的上下文，以最小的沟通成本获得可用的输出。

**关键原则**：
- AI是执行者，你是决策者——每个阶段结束后必须有人工确认
- 上下文与需求描述——给AI的信息越准确，输出质量越高
- 阶段产物要留存——需求文档、架构图、调试日志都是资产
- 工具专项专用——对话型AI做设计，代码型AI做实现

---

## 五阶段工作流

### Phase 1：需求描述（从模糊想法到可执行指令）

**三步走**：
1. **模糊需求澄清** → 用对话把需求想清楚
2. **技术选型** → 根据约束选出技术方案
3. **结构化确认** → 用模板固化结论

**工具**：Claude/ChatGPT（需求澄清）、Notion/飞书（需求存档）

**输出物**：
- 能用2-3句话描述"谁/场景/解决什么问题"
- 技术栈已选定（语言、运行方式、核心依赖）
- 标准需求文档（含系统背景、本次目标、用户场景、输入输出、边界约束、异常处理、验收标准）

---

### Phase 2：架构设计（写代码前确定系统结构）

**目标**：确定模块怎么分、数据怎么流、接口怎么定义

**工具**：
- Claude：架构方案讨论
- Cursor：项目结构生成
- Mermaid/draw.io：数据流图
- Markdown：接口约定存档

**标准步骤**：
1. 粘贴需求文档，要求AI给出项目结构和分层方案
2. 让AI解释每一层职责，确认无职责混用
3. 输出Mermaid数据流图，人工检查
4. 定义各模块接口（函数签名/类定义），不写实现
5. 人工审查并冻结架构

**输出物**：
- 项目目录结构（到文件级别）
- 数据流图
- 模块接口定义（不含实现）
- 架构文档已提交仓库

---

### Phase 3：代码生成（逐模块填充实现）

**核心陷阱**：不要把整个项目一次性丢给AI，上下文窗口有限，AI会"脑补"

**正确做法**：每次只生成一个模块，给足该模块需要的上下文，验收后再进行下一个

**工具**：
- Cursor：主力编码工具
- Claude：复杂逻辑设计
- GitHub Copilot：行级补全
- Claude Code：终端代码助手（多文件联动）

**生成顺序原则**：
- 先生成基础层（utils、models、storage），再生成业务层
- 有依赖关系的模块，先生成被依赖的一方
- 每个模块生成后立即做最小验收

**代码生成Prompt结构**：
```
[项目背景 - 一句话]
[当前任务 - 模块名、文件路径]
[该模块的职责]
[依赖的接口]
[该模块对外暴露的接口]
[数据结构]
[约束 - 不要修改其他模块、异常处理、日志格式]
```

---

### Phase 4：调试（出错不可怕，说清楚是关键）

**基本调试循环（四步走）**：
1. **把错误完整地告诉AI** — 完整报错信息、操作描述、期望vs实际、已尝试方法
2. **让AI解释原因** — 用大白话解释错误含义、最可能的问题、几种可能原因
3. **按步骤操作，反馈结果** — 逐步执行，每步完成后告诉AI结果
4. **问题解决后，记录一句话** — 「问题：___；原因：___；解决方法：___」

**重要原则**：
- **不要截图**，粘贴文本
- **不要只粘最后一行**，给完整报错
- **同一问题超过3轮没解决，立刻开新对话**

---

### Phase 5：持续迭代（项目上线不是终点）

**三种迭代场景，三个切入点**：

| 场景 | 切入点 | 说明 |
|------|--------|------|
| **新增功能** | 从Phase 1开始 | 当作独立小项目，走完整流程 |
| **性能/体验优化** | 从Phase 4开始 | 直接描述问题让AI改 |
| **代码结构混乱** | 从Phase 2开始 | 重新梳理架构，重构和加功能不要同时做 |

**最重要的一件事**：带上历史文档（需求文档、架构文档、问题记录）

## 工具速查表

| 阶段 | 核心工具 | 输入 | 输出 |
|------|----------|------|------|
| Phase 1 需求描述 | Claude, Notion | 一句话模糊想法 | 结构化需求文档、技术栈决策、验收标准 |
| Phase 2 架构设计 | Claude, Cursor, Mermaid | 需求文档、技术栈 | 目录结构、数据流图、接口约定 |
| Phase 3 代码生成 | Cursor, Claude Code | 架构文档、接口约定 | 各模块实现代码、基础测试 |
| Phase 4 调试 | Claude | 完整报错信息、操作描述 | 原因解释、逐步解决方案、问题记录 |
| Phase 5 持续迭代 | Claude | 改动类型判断 | 切入Phase选择、历史文档复用、更新文档 |

---

## 关键原则总结

1. **每个阶段有明确的输出物**，不输出不进入下一阶段
2. **Phase 1必须走完三步**：澄清→选型→结构化，不要跳步骤
3. **AI生成的每份内容都必须经过人工确认**，不自动流转
4. **上下文比prompt技巧更重要**——给AI准确的材料
5. **调试时给完整信息**，不截图，不只粘最后一行
6. **同一问题超过3轮没解决，立刻开新对话重头说**

> *规范化不是束缚，是让AI协作持续可靠的保障*

## 对满意解的启示

### 1. 工具开发参考
当满意解进入"将方法论产品化"阶段（QPMS系统、在线评估工具等），可直接应用此工作流：
- **Phase 1**：将"六维评估框架"转化为AI可执行的结构化指令
- **Phase 2**：QPMS系统的模块边界、数据流向设计
- **Phase 3**：用Cursor/Claude Code实现各评估模块

### 2. 方法论迁移
- "AI是执行者，你是决策者" → 满意解"AI辅助决策，人类最终决策"
- "上下文比prompt技巧更重要" → 强调完整背景信息收集
- "阶段产物要留存" → 满意解文档化资产沉淀

### 3. 工具链储备
| 工具 | 满意解适用场景 |
|------|---------------|
| Claude | 六维评估模型设计 |
| Cursor | QPMS系统开发 |
| Mermaid | 决策流程可视化 |
| Notion/飞书 | 满意解文档库 |
