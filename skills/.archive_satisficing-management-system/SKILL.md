## 执行摘要

本Skill建立满意解可执行的管理规则体系，确保管理不是纸上谈兵，而是有自动化保障的执行系统。

**核心原则**:
- 自动化优先：能自动的绝不人工
- 渐进式执行：不一次性重构，而是逐步迁移
- 容错机制：异常时有人工介入路径
- 可追溯：所有变更都有日志

## 一、文件管理规则

### 1.1 架构原则（第一性原则+PARA方法）

#### 第一性原则框架

| 原则 | 含义 | 执行标准 | 验收方式 |
|------|------|----------|----------|
| **单一职责** | 每个文件只有一个明确目的 | 文件名与内容100%匹配 | 自动化命名检查 |
| **无重复** | 同一内容只存一份 | MD5去重，零重复 | 每日重复扫描 |
| **无垃圾** | 临时文件及时清理 | 临时文件7天内清理 | 每周清理报告 |
| **清晰层级** | 最多4层目录深度 | 目录深度≤4 | 每日层级扫描 |
| **可追溯** | 版本历史完整 | Git提交记录完整 | 自动化Git检查 |

#### PARA分类法

```
workspace/
├── 📦 01-PROJECTS/        # 有明确截止日期的项目
├── 🌐 02-AREAS/           # 需要长期维护的责任领域
├── 📚 03-RESOURCES/       # 参考材料与资源库
└── 🗃️ 04-ARCHIVES/        # 已完成项目/历史归档
```

**触发条件**:
- 每日06:00自动检查PARA分类合规性
- 新文件创建时自动提示分类
- 每周生成PARA合规报告

**验收标准**:
- 100%文件归属明确类别
- 无"待分类"文件超过24小时
- 归档文件30天内自动压缩

### 1.2 目录层级规则

#### 4层限制强制执行

```
第1层: 类别 (01-PROJECTS/)
├── 第2层: 项目/领域 (PROJECT-001-品牌升级/)
│   ├── 第3层: 子目录 (DELIVERABLES/)
│   │   └── 第4层: 文件 (brand-guide-v1.0.md)
```

**违规处理**:
1. 检测: 自动化脚本每日扫描
2. 警告: 生成违规报告，发送通知
3. 自动修复: 提供扁平化建议
4. 人工确认: 执行前需人工确认

#### 每个目录必须包含

| 文件 | 用途 | 自动化生成 |
|------|------|------------|
| `README.md` | 目录索引与说明 | ✅ 自动生成 |
| `.meta.json` | 元数据记录 | ✅ 自动生成 |
| 可选: `CHANGELOG.md` | 变更历史 | 手动维护 |

### 1.3 文件命名规范

#### 命名格式标准

| 类型 | 格式 | 示例 | 禁止 |
|------|------|------|------|
| 文档 | 大驼峰+版本 | `BrandStrategyV1.0.md` | emoji, 中文序号 |
| 代码 | 小写+下划线 | `file_analyzer.py` | 大写, 空格 |
| 配置 | 小写+连字符 | `app-config.json` | 下划线 |
| 临时 | 日期前缀+描述 | `20260319_temp_data.json` | 纯描述 |
| 归档 | ARCHIVE_原文件名 | `ARCHIVE_BrandStrategyV0.9.md` | 其他前缀 |

#### 版本命名规范

```
V主版本.次版本.修订版本-状态

V1.0.0-stable    # 稳定版
V1.1.0-beta      # 测试版  
V2.0.0-draft     # 草稿版
```

**自动化检查**:
```bash
# 每日06:30执行命名规范检查
/scripts/management_automation/daily_check.sh naming
```

### 1.4 重复文件管理

#### 去重策略

| 优先级 | 策略 | 执行频率 | 自动化程度 |
|--------|------|----------|------------|
| P0 | MD5完全匹配检测 | 每日 | 全自动 |
| P1 | 文件名相似检测 | 每周 | 半自动 |
| P2 | 内容相似度检测 | 每月 | 人工辅助 |

#### 去重流程

```
检测 → 标记 → 人工确认 → 删除/归档 → 建立链接 → 记录日志
```

**保留规则**:
1. 保留最新修改版本
2. 保留在"正确位置"的版本
3. 删除前建立符号链接（防断链）
4. 所有删除记录在`logs/deduplication.log`

## 二、记忆管理规则

### 2.1 记忆分类体系

#### 三层记忆结构

```
memory/
├── 📝 SHORT-TERM/         # 短期记忆 (7天)
│   ├── daily/             # 每日记录
│   └── temp/              # 临时记录
├── 📚 MID-TERM/           # 中期记忆 (90天)
│   ├── weekly/            # 每周总结
│   └── monthly/           # 月度回顾
└── 🧠 LONG-TERM/          # 长期记忆 (永久)
    ├── MEMORY.md          # 核心记忆
    ├── decisions/         # 决策记录
    └── lessons/           # 经验教训
```

**触发条件**:
- 每日会话结束 → 更新SHORT-TERM
- 每周日23:00 → 生成MID-TERM摘要
- 每月最后一日 → 归档到LONG-TERM

### 2.2 记忆更新规则

#### 自动捕获内容

| 场景 | 自动捕获 | 存储位置 | 保留期限 |
|------|----------|----------|----------|
| 用户决策 | 决策内容+上下文 | decisions/ | 永久 |
| 错误教训 | 错误描述+解决方案 | lessons/ | 永久 |
| 重要信息 | 用户偏好/约束 | MEMORY.md | 永久 |
| 每日对话 | 要点摘要 | daily/ | 7天 |

#### 记忆整理流程

```
每日自动生成 → 每周人工审查 → 每月归档整合 → 每季清理优化
```

**人工介入时机**:
- 每周日: 审查本周记忆摘要
- 每月1日: 确认上月归档内容
- 异常时: 记忆冲突/丢失时人工介入

### 2.3 记忆完整性保障

#### 备份策略

| 类型 | 频率 | 位置 | 保留期 |
|------|------|------|--------|
| 实时备份 | 每次写入 | .backup/memory/ | 24小时 |
| 每日备份 | 每日06:00 | backup/daily/ | 7天 |
| 每周备份 | 每周日 | backup/weekly/ | 4周 |
| 每月备份 | 每月1日 | backup/monthly/ | 12月 |

#### 完整性验证

```bash
# 每日执行记忆完整性验证
/scripts/management_automation/integrity_validator.sh memory
```

**验证项目**:
- [ ] 每日记忆文件存在且非空
- [ ] MEMORY.md可正常读取
- [ ] 备份文件完整性
- [ ] 无孤立记忆片段

## 三、任务管理规则

### 3.1 任务生命周期

#### 任务状态流转

```
CREATED → ASSIGNED → IN-PROGRESS → REVIEW → COMPLETED
            ↓              ↓            ↓
         BLOCKED      ON-HOLD      REJECTED
            ↓              ↓            ↓
         RESOLVED    RESUMED      ARCHIVED
```

**状态定义**:

| 状态 | 含义 | 停留时限 | 超时处理 |
|------|------|----------|----------|
| CREATED | 已创建待分配 | 24小时 | 自动提醒分配 |
| ASSIGNED | 已分配待执行 | 48小时 | 升级提醒 |
| IN-PROGRESS | 执行中 | 按预估 | 进度检查 |
| BLOCKED | 被阻塞 | 24小时 | 升级处理 |
| REVIEW | 待审核 | 24小时 | 提醒审核人 |
| COMPLETED | 已完成 | - | 归档 |

### 3.2 任务优先级规则

#### 优先级定义

| 级别 | 标识 | 响应时间 | 执行时限 | 升级阈值 |
|------|------|----------|----------|----------|
| P0 | 🔴 紧急 | 立即 | 4小时 | 2小时 |
| P1 | 🟠 高 | 2小时 | 24小时 | 12小时 |
| P2 | 🟡 中 | 4小时 | 72小时 | 48小时 |
| P3 | 🟢 低 | 24小时 | 7天 | 5天 |
| P4 | ⚪ 待排 | 无 | 无 | - |

#### 自动化任务监控

```bash
# 每小时检查任务状态
/scripts/management_automation/daily_check.sh tasks
```

**监控内容**:
- P0/P1任务执行状态
- 阻塞任务原因分析
- 即将超时任务预警
- 长期未完成任务清理

### 3.3 任务交付标准

#### 交付检查清单

| 检查项 | 标准 | 自动化 | 人工确认 |
|--------|------|--------|----------|
| 文件完整性 | 所有指定文件存在 | ✅ | - |
| 格式合规 | 符合命名/格式规范 | ✅ | - |
| 内容完整 | 达到最低内容要求 | - | ✅ |
| 引用有效 | 所有链接/引用可访问 | ✅ | - |
| 版本标记 | 明确版本号 | ✅ | - |

#### 交付物自动验证

```
提交 → 自动检查 → 结果通知 → 人工确认 → 归档/打回
```

## 四、安全管理规则

### 4.1 敏感信息管理

#### 敏感信息分类

| 级别 | 类型 | 示例 | 处理方式 |
|------|------|------|----------|
| S0 | 机密 | 密码、密钥 | 加密存储，环境变量 |
| S1 | 私密 | 个人信息 | 权限控制，脱敏展示 |
| S2 | 内部 | 内部策略 | 访问控制，水印标记 |
| S3 | 公开 | 公开文档 | 无需特殊处理 |

#### 存储规则

```
secrets/
├── encrypted/             # 加密文件 (S0)
│   ├── credentials.enc
│   └── api-keys.enc
├── restricted/            # 限制访问 (S1)
│   └── personal/
└── internal/              # 内部文件 (S2)
    └── strategies/
```

**自动化检查**:
```bash
# 每日检查敏感信息泄露风险
/scripts/management_automation/daily_check.sh security
```

### 4.2 访问控制规则

#### 目录权限矩阵

| 目录 | 所有者 | 组 | 其他 | 特殊规则 |
|------|--------|-----|------|----------|
| secrets/ | 600 | - | - | 加密存储 |
| 01-PROJECTS/ | 755 | 750 | - | 项目隔离 |
| 04-ARCHIVES/ | 755 | 755 | - | 只读 |
| 05-TEMP/ | 777 | - | - | 自动清理 |

### 4.3 安全审计

#### 审计日志内容

| 事件 | 记录内容 | 保留期 |
|------|----------|--------|
| 文件访问 | 用户+时间+文件+操作 | 90天 |
| 配置变更 | 变更前后+变更人+时间 | 180天 |
| 权限变更 | 权限+变更+授权人 | 永久 |
| 安全事件 | 事件详情+处理结果 | 永久 |

## 五、执行自动化机制

### 5.1 自动化脚本体系

#### 脚本分类

| 脚本 | 频率 | 功能 | 输出 |
|------|------|------|------|
| `daily_check.sh` | 每日06:00 | 全面日检 | 日报 |
| `weekly_audit.sh` | 每周日23:00 | 深度审计 | 周报 |
| `monthly_cleanup.sh` | 每月1日03:00 | 月度清理 | 清理报告 |
| `integrity_validator.sh` | 按需 | 完整性验证 | 验证报告 |

#### 执行流程

```
定时触发 → 前置检查 → 执行脚本 → 生成报告 → 发送通知 → 记录日志
```

### 5.2 自动化触发条件

#### 定时触发

| 任务 | 时间 | 条件 | 容错 |
|------|------|------|------|
| 日检 | 06:00 | 系统空闲 | 延迟1小时重试 |
| 周审 | 周日23:00 | 日检正常 | 周一03:00补跑 |
| 月清 | 1日03:00 | 周审完成 | 人工触发 |

#### 事件触发

| 事件 | 触发动作 | 响应时间 |
|------|----------|----------|
| 新文件创建 | 分类检查 | 实时 |
| 大文件上传 | 存储检查 | 实时 |
| 敏感操作 | 安全审计 | 实时 |
| 错误日志 | 告警通知 | 5分钟 |

### 5.3 人工介入机制

#### 自动升级规则

| 场景 | 自动处理 | 升级条件 | 人工介入 |
|------|----------|----------|----------|
| 文件冲突 | 标记待解决 | 2小时内未解决 | 通知负责人 |
| 重复文件 | 提供建议 | 建议未被采纳 | 每周汇总 |
| 安全告警 | 记录日志 | 高危告警 | 立即通知 |
| 任务阻塞 | 分析原因 | 24小时未解 | 升级处理 |

#### 人工介入路径

```
发现问题 → 查看报告 → 决策处理 → 记录原因 → 更新规则
```

### 5.4 可追溯体系

#### 变更追踪

| 变更类型 | 追踪内容 | 存储位置 |
|----------|----------|----------|
| 文件变更 | 修改前后+修改人+时间 | Git历史 |
| 配置变更 | 配置项+变更+影响 | config/history/ |
| 规则变更 | 规则版本+变更说明 | rules/changelog.md |
| 权限变更 | 权限+授权+原因 | audit/permissions/ |

#### 日志规范

```
[YYYY-MM-DD HH:MM:SS] [LEVEL] [MODULE] [ACTION] [DETAILS] [RESULT]

示例:
[2026-03-19 06:00:15] [INFO] [FILE] [DAILY_CHECK] 扫描1200个文件 [PASS]
[2026-03-19 06:05:22] [WARN] [FILE] [DUPLICATE_FOUND] 发现3个重复文件 [PENDING]
```
