# 企业微信文档同步技能 (WeCom Doc Sync)

## 核心原则

> **同步后必须完整性检查**
> 
> 企业微信文档不支持完整的Markdown格式（如复杂表格、嵌套列表），同步后内容可能不完整。
> 必须建立完整性检查流程，确保核心信息无遗漏。

## 同步流程

### Step 1: 准备内容

- 确保本地文档已最终定稿
- 检查文档大小（企业微信有单文档大小限制）
- 复杂内容考虑分段同步

### Step 2: 执行同步

使用 `wecom_mcp` 工具的 `doc` 类别：
- `create_doc` - 创建新文档
- `edit_doc_content` - 编辑已有文档

### Step 3: 完整性检查（关键！）

同步后，必须在企业微信中打开文档，人工检查以下内容：

#### 必检清单

| 检查项 | 检查方法 | 合格标准 |
|--------|----------|----------|
| **标题层级** | 浏览文档结构 | 标题层级正确显示 |
| **表格内容** | 检查所有表格 | 表格完整，无截断 |
| **列表层级** | 检查嵌套列表 | 层级关系正确 |
| **关键数据** | 核对数字、日期 | 无遗漏或错误 |
| **链接** | 点击测试 | 链接可正常打开 |
| **特殊字符** | 检查符号、公式 | 显示正常 |

#### 常见问题

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 表格截断 | 企业微信表格列数限制 | 简化表格，或拆分为多个表格 |
| 嵌套列表混乱 | Markdown解析差异 | 减少层级，或使用简单列表 |
| 特殊符号丢失 | 编码问题 | 替换为文字描述 |
| 内容超长 | 单文档大小限制 | 分段同步，创建多个文档 |

### Step 4: 修复与确认

- 发现问题 → 在本地修复 → 重新同步 → 再次检查
- 确认无误 → 更新文档索引 → 通知相关人员

## 分段同步策略

对于超长文档（如百科全书），建议分段同步：

```
原文档：满意解研究所_V1.2_增强版.md (13000字)

分段方案：
├── 卷一：惟吾德馨（创始人背景）→ 单独文档
├── 卷二：期待的伙伴（服务对象）→ 单独文档
├── 卷三：五路图腾（方法论）→ 单独文档
├── 卷四-六：专家网络+产品+工具 → 单独文档
├── 卷七-十一：组织+营销+战略 → 单独文档
└── 附录：修订说明 → 单独文档

索引文档：满意解研究所_V1.2_完整索引
```

## 完整性检查脚本（建议）

```bash
#!/bin/bash
# 完整性检查脚本示例

echo "=== 企业微信文档完整性检查 ==="
echo ""
echo "请人工检查以下项目："
echo ""
echo "□ 1. 打开企业微信文档，浏览标题结构"
echo "□ 2. 检查所有表格是否完整"
echo "□ 3. 核对关键数据（数字、日期、金额）"
echo "□ 4. 测试所有链接是否正常"
echo "□ 5. 检查特殊字符显示"
echo "□ 6. 确认文档末尾无截断"
echo ""
echo "如发现问题，请："
echo "  1. 记录问题详情"
echo "  2. 在本地修复"
echo "  3. 重新同步"
echo "  4. 再次检查"
echo ""
read -p "检查完成，是否确认无问题？(y/n) " confirm

if [ "$confirm" = "y" ]; then
    echo "✅ 完整性检查通过"
    # 更新索引、通知相关人员
else
    echo "❌ 请修复问题后重新检查"
fi
```

## 与飞书的对比

| 功能 | 飞书 | 企业微信 | 建议 |
|------|------|----------|------|
| 文档同步 | ✅ 完整 | ⚠️ 需检查 | 企业微信同步后必须检查 |
| 表格支持 | ✅ 完整 | ⚠️ 有限制 | 简化表格设计 |
|  Markdown | ✅ 完整 | ⚠️ 部分支持 | 避免复杂语法 |
| 实时协作 | ✅ 完整 | ✅ 完整 | 均可 |

## 最佳实践

1. **先本地定稿**：不在企业微信上直接编辑
2. **简化格式**：使用简单的Markdown语法
3. **分段同步**：超长文档分卷同步
4. **必做检查**：同步后必须人工检查
5. **建立索引**：多文档时创建索引文档
6. **定期核对**：重要文档定期与企业微信版本核对

## 紧急备用

如企业微信同步失败或格式混乱：
- 飞书作为紧急备用通道
- 本地Markdown文件为最终源头
- 定期备份到GitHub
