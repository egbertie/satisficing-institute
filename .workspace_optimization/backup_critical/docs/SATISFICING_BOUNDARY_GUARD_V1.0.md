## 一、为什么需要边界守护

### 1.1 范围蔓延的必然性

**创始人面临的持续诱惑**：
- 客户说"能不能顺便帮我看看这个"
- 朋友说"你这个也可以做XXX"
- 自己觉得"这个好像也能试试"
- 担心"不做这个会不会错过机会"

**范围蔓延的后果**：
- 资源分散，核心目标无法达成
- 精力耗散，服务质量下降
- 边界模糊，品牌定位混乱
- 团队疲惫（即使是OPC）

### 1.2 边界守护的目标

**不是**为了限制可能性，**而是**为了保护专注力。

**核心原则**：在OPC形态下，我们只能做好一件事。把这件事做到极致，比做十件平庸的事更有价值。

## 二、决策检查清单

### 2.1 新机会评估清单

当新的机会/建议出现时，用以下清单评估（基于 [SCOPE V1.0 第五章](SATISFICING_SCOPE_V1.0.md#五边界决策流程图) 的边界决策流程）：

```
┌─────────────────────────────────────────────────────────┐
│                 新机会评估清单                          │
├─────────────────────────────────────────────────────────┤
│ □ 是否属于"合伙人选择决策"相关？                        │
│   如果否 → 不做                                          │
├─────────────────────────────────────────────────────────┤
│ □ 必须由Egbertie亲自做吗？                              │
│   如果否 → 查看合作清单                                  │
├─────────────────────────────────────────────────────────┤
│ □ OPC资源能否支撑？                                     │
│   如果否 → 不做                                          │
├─────────────────────────────────────────────────────────┤
│ □ 是否有助于验证核心商业模式？                          │
│   如果否 → 不做                                          │
├─────────────────────────────────────────────────────────┤
│ □ 现在不做，会影响核心目标吗？                          │
│   如果否 → 列入"未来考虑"                               │
├─────────────────────────────────────────────────────────┤
│ □ 有客户明确付费意愿吗？                                │
│   如果否 → 不做（验证期不做假设）                        │
├─────────────────────────────────────────────────────────┤
│ □ 投入产出比高于核心服务吗？                            │
│   如果否 → 不做                                          │
└─────────────────────────────────────────────────────────┘
```

**通过标准**：全部打勾才能做。

### 2.2 客户请求评估清单

当客户提出边界外的请求时：

```
┌─────────────────────────────────────────────────────────┐
│                 客户请求评估清单                        │
├─────────────────────────────────────────────────────────┤
│ □ 这个请求是否与合伙人选择直接相关？                    │
│   如果否 → 礼貌拒绝，解释边界                            │
├─────────────────────────────────────────────────────────┤
│ □ 这个请求是否在核心服务范围内？                        │
│   如果是 → 正常交付                                      │
├─────────────────────────────────────────────────────────┤
│ □ 这个请求能否通过合作方解决？                          │
│   如果是 → 引荐合作方                                    │
├─────────────────────────────────────────────────────────┤
│ □ 这个请求是否紧急且重要？                              │
│   如果是 → 评估是否调整边界（例外）                      │
├─────────────────────────────────────────────────────────┤
│ □ 这个请求是否具有普遍性？                              │
│   如果是 → 列入"未来产品清单"                           │
└─────────────────────────────────────────────────────────┘
```

**标准回应话术**：
> "感谢您的信任！这个需求很重要，但不在我们第一年的服务范围内。我可以为您引荐一位专业的[合作方类型]，他们更擅长这个领域。"

### 2.3 自建 vs 合作 vs 不做决策树

```
新需求/新机会
      ↓
【第一层】是否属于核心能力？
      ↓
     是 → 自己做
      ↓
     否 → 【第二层】能否找到合作方？
              ↓
          否 → 【第三层】是否必须做？
                  ↓
              否 → 不做
                  ↓
              是 → 重新评估核心能力边界
                  ↓
          是 → 合作做
```

## 三、月度边界审视机制

### 3.1 审视时间与形式

| 项目 | 说明 |
|-----|------|
| **时间** | 每月最后一个工作日下午 |
| **时长** | 30-60分钟 |
| **形式** | 自我复盘，可请AI助手协助 |
| **输出** | 边界审视记录（写入memory） |

### 3.2 审视问题清单

**第一部分：回顾**

1. 本月完成了哪些案例？
2. 本月有哪些超出边界的行为？
3. 本月拒绝了哪些"诱惑"？
4. 本月的时间分配是否符合优先级？

**第二部分：检查**

5. 当前的业务范围是否需要调整？
   - 收紧：发现资源比预期更紧张
   - 放宽：核心目标已提前达成
   - 不变：维持现状

6. 不做清单上的事，是否有必须做的？
   - 如果有，原因是什么？
   - 是否只是"诱惑"而非"必须"？

7. 合作方是否满足需求？
   - 是否需要新增合作方？
   - 是否需要淘汰合作方？

**第三部分：展望**

8. 下月最大的"诱惑"可能是什么？
9. 下月需要特别警惕的边界风险是什么？
10. 需要调整哪些机制？

### 3.3 审视决策矩阵

| 情况 | 决策 |
|-----|------|
| 核心目标进度正常，无边界突破 | 维持现状 |
| 核心目标进度正常，有边界突破 | 收紧边界，回归正轨 |
| 核心目标进度落后，无边界突破 | 分析原因，优化执行 |
| 核心目标进度落后，有边界突破 | 立即止损，回归核心 |
| 核心目标提前达成 | 可考虑适度放宽边界 |

### 3.4 审视记录模板

```markdown
# 边界审视记录 - YYYY年MM月

## 本月概况
- 完成案例数：X
- 超出边界行为：Y
- 拒绝的诱惑：Z

## 边界检查
- [ ] 范围是否合适？
- [ ] 不做清单是否遵守？
- [ ] 合作方是否满足需求？

## 调整决策
- 调整类型：维持/收紧/放宽
- 调整内容：XXX
- 生效时间：立即/下月

## 下月警惕
- 可能的诱惑：XXX
- 应对措施：XXX
```

## 四、边界突破的例外机制

### 4.1 什么情况下可以突破边界

**原则**：边界可以突破，但必须是有意识的决策，而非无意识的蔓延。

**可以突破的情况**：

| 情况 | 条件 | 审批 |
|-----|------|------|
| 多个客户明确要求某项服务 | >3个客户，且愿意付费 | 月度审视决定 |
| 资源约束发生根本变化 | 获得融资/全职加入 | 立即调整规划 |
| 市场环境剧变 | 如疫情导致无法面谈 | 针对性调整 |
| 核心假设被证伪 | 商业模式不成立 | 立即止损或转型 |

### 4.2 突破边界的决策流程

```
突破需求出现
      ↓
是否属于上述4种情况？
      ↓
     否 → 坚持不做
      ↓
     是 → 填写《边界突破申请表》
              ↓
          等待月度审视（紧急事项立即处理）
              ↓
          审视通过 → 调整边界
              ↓
          审视不通过 → 维持原边界
```

### 4.3 边界突破申请表（模板）

```
┌─────────────────────────────────────────────────────────┐
│                 边界突破申请表                          │
├─────────────────────────────────────────────────────────┤
│ 申请日期：YYYY-MM-DD                                    │
│ 申请事项：XXX                                           │
├─────────────────────────────────────────────────────────┤
│ 突破类型：                                              │
│ □ 多个客户要求  □ 资源变化  □ 环境变化  □ 假设证伪      │
├─────────────────────────────────────────────────────────┤
│ 突破原因：                                              │
│ （详细说明为什么必须突破）                               │
├─────────────────────────────────────────────────────────┤
│ 不突破的后果：                                          │
│ （详细说明如果不突破会怎样）                             │
├─────────────────────────────────────────────────────────┤
│ 突破后的影响：                                          │
│ 对核心目标的影响：XXX                                   │
│ 对资源的影响：XXX                                       │
│ 对品牌的影响：XXX                                       │
├─────────────────────────────────────────────────────────┤
│ 申请决策：                                              │
│ □ 批准    □ 拒绝    □ 暂缓                              │
```

## 五、常见诱惑应对话术

### 5.1 应对客户请求

**客户："能不能帮我做股权设计？"**
> "股权设计需要专业的法律背景，我可以为您引荐一位专门做早期企业股权的律师，他们更专业。我的专注点是帮您判断合伙人是否合适，这是两个不同的事。"

**客户："能不能长期陪跑？"**
> "长期陪跑需要全职投入，我目前的模式是专注于合伙人选择的评估。如果您需要长期陪伴，我可以为您引荐一位专业的创业者教练。"

**客户："能不能便宜点，后面再补？"**
> "我们的定价已经是最小可行版本了。如果预算确实紧张，建议您先自己梳理清楚需求，等时机成熟再找我。"

### 5.2 应对朋友建议

**朋友："你们也可以做培训课程啊，这个规模大。"**
> "培训确实规模更大，但第一年我只想专注把评估服务做好。等案例积累多了，如果有需求再考虑。"

**朋友："你应该做个APP，这样更方便。"**
> "客户要的是我的判断，不是APP。第一年先用手工方式跑通流程，技术投入以后再说。"

### 5.3 应对自我怀疑

**内心："不做这个会不会错过机会？"**
> 回应：错过的机会不可怕，可怕的是分散精力导致核心目标失败。专心做好一件事，机会自然来。

**内心："别人都在做，我是不是也该做？"**
> 回应：别人的资源、阶段、能力都不一样。按自己的节奏走，不跟风。

**内心："只做这个会不会太局限？"**
> 回应：第一年的"局限"是为了第二年的"扩展"。没有验证的成功，扩展就是冒险。

## 七、边界守护宣言

> 满意解第一年的边界，不是为了限制可能性，而是为了保护专注力。
>
> 在OPC的约束下，我们只能做好一件事。把这件事做到极致，比做十件平庸的事更有价值。
>
> 边界是护城河，不是牢笼。
>
> 说"不"是为了更好地说"是"。
>
> 少即是多，慢即是快。
>
> 第一年，只做合伙人评估这一件事。
