## 一、架构设计

### 1.1 核心原则

```
┌─────────────────────────────────────────────────────────────┐
│                    设计原则                                 │
├─────────────────────────────────────────────────────────────┤
│  1. 零定期消耗 — 没有固定检查，Token只在必要时使用          │
│  2. 事件即信号 — 状态变化触发，而非时间驱动                 │
│  3. 人工即主权 — 所有执行需用户确认，系统是顾问而非代理     │
│  4. 静默即美德 — 无异常时保持静默，不打扰用户               │
└─────────────────────────────────────────────────────────────┘
```

## 二、事件定义

### 2.1 事件类型

```yaml
# 任务事件
task_events:
  - name: task_created
    description: 新任务创建
    priority: low
    action: 记录，如用户离线则准备通知
    
  - name: task_completed
    description: 任务完成
    priority: medium
    action: 检查是否有待补位任务
    
  - name: task_blocked
    description: 任务阻塞（超过30分钟无进展）
    priority: high
    action: 分析原因，准备建议报告
    
  - name: task_timeout
    description: 任务超时
    priority: urgent
    action: 立即通知，生成处理建议

# Token事件
token_events:
  - name: token_low_warning
    description: Token余量<30%
    priority: medium
    action: 生成节约建议
    
  - name: token_critical
    description: Token余量<10%
    priority: urgent
    action: 紧急通知，暂停非必要任务
    
  - name: token_budget_changed
    description: Token预算调整
    priority: low
    action: 更新策略参数

# 用户状态事件
user_events:
  - name: user_went_offline
    description: 用户离线
    priority: medium
    action: 标记可补位状态，准备夜间模式
    
  - name: user_came_online
    description: 用户上线
    priority: low
    action: 检查离线期间状态，提供摘要
    
  - name: night_mode_activated
    description: 夜间模式激活
    priority: low
    action: 激活预设白名单自动执行
```

### 2.2 事件优先级矩阵

| 事件 | 优先级 | 通知方式 | 响应时间 |
|------|--------|---------|---------|
| 任务超时 | 🔴 紧急 | 立即通知 | <1分钟 |
| Token<10% | 🔴 紧急 | 立即通知 | <1分钟 |
| 任务阻塞 | 🟡 高 | 10分钟内通知 | <10分钟 |
| Token<30% | 🟡 高 | 下次交互时提示 | <1小时 |
| 用户离线 | 🟢 中 | 静默标记 | 无需响应 |
| 任务完成 | 🟢 中 | 夜间摘要 | <2小时 |
| Token预算调整 | ⚪ 低 | 静默记录 | 无需响应 |

### 4.2 夜间模式白名单

```yaml
# 夜间模式配置
night_mode:
  enabled: false  # 默认关闭，需用户主动开启
  
  # 自动执行白名单（用户预设）
  auto_execute_whitelist:
    - 数据整理类任务
    - 报告生成类任务
    - 文件归档类任务
    
  # 需要确认的任务（即使夜间模式也等待确认）
  always_confirm:
    - 涉及外部通信的任务
    - 消耗Token > 1000的任务
    - 删除/修改类操作
    
  # 夜间模式边界
  constraints:
    max_token_per_night: 3000  # 每晚最大Token消耗
    max_concurrent_tasks: 3     # 最大并行任务数
    auto_shutdown_time: "06:00" # 自动退出夜间模式时间
```

## 七、风险控制

### 7.1 风险清单

| 风险ID | 风险描述 | 可能性 | 影响 | 缓解措施 |
|--------|---------|--------|------|---------|
| R01 | 用户忘记检查导致任务堆积 | 中 | 高 | 智能提醒（非强制） |
| R02 | 事件监听失败导致遗漏 | 低 | 高 | 健康检查机制 |
| R03 | 夜间模式过度消耗Token | 低 | 中 | 预算上限控制 |
| R04 | 用户不熟悉新指令 | 中 | 低 | 帮助文档+示例 |
| R05 | 紧急事件响应延迟 | 低 | 高 | 关键事件自动通知 |

### 7.2 回滚方案

```
如果方案C运行不稳定：

阶段1: 启用降级模式
├── 临时启用方案B（分层架构）
├── 保留事件监听
└── 增加定期检查作为保险

阶段2: 完全回滚
├── 恢复原有4个Cron任务
├── 保留新的事件监听作为补充
└── 分析问题后重新设计
```
