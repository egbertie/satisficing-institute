# 零空置强制执行机制 V2.0 - 制度即代码

**创建时间**: 2026-03-15 07:50  
**触发原因**: 用户多次反馈"进行中却全部阻塞 = 虚假忙碌"反复发生  
**核心原则**: 制度不是写入文档，而是写入代码强制执行

---

## 一、问题诊断：为什么反复发生

### 历史记录
| 时间 | 问题 | 用户反馈 |
|------|------|----------|
| 2026-03-15 07:03 | 任务协调检查反复询问已解决问题 | "我回答无数遍这两个问题了" |
| 2026-03-15 07:30 | 进行中全部阻塞 | "资源就空置在这里" |
| 2026-03-15 07:46 | 再次强调 | "反复发生...结果还是会发生" |

### 根本原因分析
**不是用户指令不清，是机制设计错误**：

| 层级 | 旧机制 | 问题 |
|------|--------|------|
| 规则 | 写在HEARTBEAT.md | 文档级，依赖人记住 |
| 检查 | 每小时cron | 频率低，发现滞后 |
| 补位 | 人工决策 | 延迟大，需要人触发 |
| 队列 | 无备用队列 | 无任务可补位 |
| 并行 | 单线推进 | 效率低 |

**本质**: 旧机制是"建议性"的，不是"强制性"的。

---

## 二、解决方案：制度即代码

### 核心认知升级
```
❌ 旧认知: "规则写入文档，大家遵守"
✅ 新认知: "规则写入代码，强制执行"
```

### Zero Idle Protocol V2.0

#### 2.1 零容忍规则（代码强制执行）

```python
# skills/zero-idle-enforcer/enforcer.py

RULE_001 = "等待外界条件 = 暂停/待补充，不得占用'进行中'"
RULE_002 = "进行中为空 = 0延迟自动从六线队列补位"
RULE_003 = "资源空置>5分钟 = 强制告警并立即激活备用队列"
RULE_004 = "任何虚假忙碌都触发强制执行"

# 强制执行触发条件
if 进行中任务全部阻塞:
    立即触发补位()  # 延迟<1秒
    向用户报告()
    记录违规()
```

#### 2.3 强制执行触发点

| 触发点 | 频率 | 执行动作 |
|--------|------|----------|
| 每次心跳 | ~30分钟 | 强制执行检查 |
| Cron Job | 每10分钟 | 强制执行检查 |
| 任务状态变更 | 实时 | 立即触发补位 |
| 用户交互间隙 | >5分钟 | 自动检查并补位 |

#### 2.4 强制执行响应

**违规类型1: 进行中全部阻塞（严重程度5/5）**
```
响应: IMMEDIATE_FILL
动作: 
  1. 立即从六线队列各激活1个任务（最少6个）
  2. 随机再激活2-4个任务
  3. 向用户报告违规和补位情况
  4. 记录违规日志
```

**违规类型2: 无真实推进任务（严重程度4/5）**
```
响应: IMMEDIATE_FILL
动作: 同上
```

**违规类型3: 任务僵死运行（严重程度3/5）**
```
响应: PARALLEL_FILL
动作:
  1. 保持原任务继续
  2. 并行启动4个新任务（六线各1个）
  3. 记录并行执行状态
```

## 三、执行监督：永不空置

### 监督机制

```
┌─────────────────────────────────────────────────────┐
│                  零空置强制执行引擎                    │
├─────────────────────────────────────────────────────┤
│  输入层: 任务状态变更 / 心跳 / Cron / 用户交互间隙      │
│           ↓                                          │
│  检查层: 是否全部阻塞？/ 是否空置？/ 是否僵死？        │
│           ↓                                          │
│  决策层: 违规类型判断 → 严重程度评估 → 响应策略选择    │
│           ↓                                          │
│  执行层: 立即补位 / 并行填充 / 告警 / 记录            │
│           ↓                                          │
│  输出层: 向用户报告 / 更新状态 / 日志记录             │
└─────────────────────────────────────────────────────┘
```

### 关键指标监控

| 指标 | 目标 | 监控频率 |
|------|------|----------|
| 真实推进任务数 | >= 1 | 实时 |
| 资源空置时间 | = 0秒 | 实时 |
| 六线并行度 | >= 4线 | 每10分钟 |
| 违规次数 | = 0 | 每10分钟 |

## 六、持续改进

### 优化方向

1. **智能补位**: 根据历史数据预测最优补位任务
2. **动态队列**: 根据项目阶段自动调整补位队列
3. **效能分析**: 分析六线并行效率，持续优化
4. **用户偏好**: 学习用户偏好，优先激活相关任务

### 反馈机制

如发现任何"进行中全部阻塞"或"资源空置"情况：
- 系统将自动记录并立即补位
- 同时向用户报告违规情况和补位行动
- 24小时内不得重复发生同类违规
