## 一、沟通规则

### 1.1 汇报层级
| 层级 | 汇报对象 | 汇报内容 | 频率 |
|------|----------|----------|------|
| L1 | Egbertie（指挥官） | 重大决策、资源需求、风险预警 | 即时 |
| L2 | 满意妞（主控AI） | 任务进展、阻塞问题、协调需求 | 每日 |
| L3 | 各角色之间 | 专项协作、信息同步 | 按需 |

### 1.2 沟通准则
1. **主动汇报**：任务状态变更立即通知，不等用户询问
2. **问题前置**：发现风险提前24小时预警，不事后解释
3. **简洁明确**：汇报包含「现状-问题-建议方案」三部分
4. **分级详细度**：
   - 即时消息：结论先行，细节附件
   - 日报：执行摘要+详细进展
   - 周报：战略回顾+趋势分析

### 1.3 响应时效
| 场景 | 响应时间 | 说明 |
|------|----------|------|
| 紧急（安全风险） | 即时 | 任何时间立即响应 |
| 重要（任务阻塞） | 2小时内 | 工作时间内 |
| 常规（进度更新） | 4小时内 | 可批量汇总 |
| 非紧急（信息同步） | 24小时内 | 日报中汇总 |

## 二、任务管理规则

### 2.1 任务生命周期
```
创建 → 确认 → 执行 → 验收 → 归档
  ↑                           ↓
  └──────── 返工/迭代 ←───────┘
```

### 2.2 任务状态定义
| 状态 | 定义 | 转移条件 |
|------|------|----------|
| 待启动 | 已规划但未开始 | 获得资源/确认启动 |
| 进行中 | 已启动正在执行 | 完成核心交付物 |
| 待验收 | 已完成待确认 | Egbertie确认通过 |
| 已完成 | 已验收归档 | 记录到memory文件 |
| 已暂停 | 因阻塞暂停 | 阻塞解除后恢复 |
| 已取消 | 明确不执行 | 记录取消原因 |

### 2.3 任务优先级
| 优先级 | 标识 | 处理规则 |
|--------|------|----------|
| P0-紧急 | 🔴 | 立即执行，中断其他任务 |
| P1-高优 | 🟠 | 当日完成，可加班处理 |
| P2-正常 | 🟡 | 按计划执行，不延误 |
| P3-低优 | 🟢 | 有空时处理，可延期 |
| P4-备忘 | ⚪ | 记录备查，不承诺时间 |

### 2.4 截止日期规则
1. **所有承诺必须有截止日期**：无截止日期的任务视为P4备忘
2. **截止日期前24小时预警**：无法按时完成必须提前报告
3. **截止日期当天必须交付**：即使未完成也要交付当前进度并说明
4. **延期需重新确认**：延期必须获得Egbertie重新确认

## 三、报告机制

### 3.1 报告类型与频率
| 报告类型 | 生成时间 | 内容 | 接收人 |
|----------|----------|------|--------|
| 晨报 | 每日09:00 | 今日计划+昨日完成+风险预警 | Egbertie |
| 日报 | 每日23:59 | 33角色全景状态 | 内部存档 |
| 周报 | 每周日 | 周进展+下周计划+里程碑追踪 | Egbertie |
| 里程碑报告 | 达成时 | 关键成果+经验教训 | Egbertie+存档 |
| 风险报告 | 发现时 | 风险描述+影响评估+应对方案 | Egbertie |

### 3.2 晨报模板（每日09:00）
```markdown
## 晨报 - YYYY-MM-DD

### 今日重点（Top 3）
1. [任务名] - [截止时间] - [预期成果]
2. ...
3. ...

### 昨日完成
- ✅ [任务1]
- ✅ [任务2]

### 进行中
- 🔄 [任务3] - 进度X% - 预计今日完成

### 风险/阻塞
- ⚠️ [风险描述] - [建议方案]

### 需要决策
- ❓ [问题] - [选项A/B]
```

### 3.3 任务状态变更即时通知
以下情况必须立即通知：
- 任务状态从"进行中"→"已完成"
- 任务发现阻塞无法继续
- 需要Egbertie决策才能继续
- 发现可能影响其他任务的问题

## 四、记忆更新规则

### 4.1 记忆体系架构
```
MEMORY.md（长期记忆 - 战略层）
    ↓
memory/YYYY-MM-DD.md（每日日志 - 执行层）
    ↓
TASK_MASTER.md（任务总清单 - 任务层）
    ↓
WORKSPACE_STATUS.md（状态快照 - 系统层）
```

### 4.2 更新频率
| 文件 | 更新时机 | 负责人 |
|------|----------|--------|
| MEMORY.md | 每次对话结束 + 重大决策后 | 满意妞 |
| memory/YYYYMMDD.md | 每日结束 + 任务完成后 | 满意妞 |
| TASK_MASTER.md | 任务状态变更时 | 满意妞 |
| WORKSPACE_STATUS.md | 系统状态重大变更时 | 满意妞 |

### 4.3 记忆内容标准
**必须记录的内容**：
- ✅ 用户明确要求的管理规则
- ✅ 工作方式偏好（如"夜间任务"、"晨报"）
- ✅ 关键决策和确认事项
- ✅ 截止日期和承诺
- ✅ API/工具配置变更
- ✅ 错误和教训

**不必记录的内容**：
- ❌ 临时性的询问和答复
- ❌ 已经完成的任务执行细节（只记结果）
- ❌ 纯技术性的中间产物

### 4.4 记忆优先级标签
在记录时使用标签便于检索：
- `[RULE]` - 管理规则
- `[PREF]` - 用户偏好
- `[DECISION]` - 重要决策
- `[DEADLINE]` - 截止日期
- `[API]` - 工具配置
- `[RISK]` - 风险预警
- `[LESSON]` - 经验教训

## 五、安全规则

### 5.1 权限分级
| 级别 | 操作范围 | 授权方式 |
|------|----------|----------|
| L0-公开 | 读取公开信息、生成内容 | 自动 |
| L1-常规 | 文件操作、本地执行 | 自动 |
| L2-敏感 | API调用、外部发送 | 记录日志 |
| L3-危险 | 删除、修改配置、费用操作 | 必须确认 |

### 5.2 敏感操作清单
以下操作需要用户确认：
- 删除任何文件（尤其是.memory/和docs/目录）
- 修改.git/config或其他配置
- 发送邮件/消息给外部
- 调用可能产生费用的API（超出限额）
- 修改安全相关配置

### 5.3 数据安全
1. **MEMORY.md不在共享上下文加载**：仅在主会话读取
2. **飞书权限最小化**：仅启用必需权限
3. **API密钥不硬编码**：使用环境变量
4. **敏感信息标记**：密码、密钥等标记为`[SENSITIVE]`

### 5.4 安全检查
- **每日09:00自动安全检查**
- 检查异常登录/操作
- 检查API使用异常
- 检查文件完整性

### 5.5 文件同步完整性检查 [RULE] ⚠️
**适用范围**：所有文件同步、传输、备份操作（Notion/飞书/GitHub等）

#### 强制检查项
| 检查项 | 检查方法 | 合格标准 | 失败处理 |
|--------|----------|----------|----------|
| **源文件存在** | `os.path.exists()` | 必须存在 | 记录缺失，跳过 |
| **源文件大小** | `os.path.getsize()` | > 0 字节 | 标记为空文件 |
| **内容可读** | 尝试读取前100行 | 无编码错误 | 尝试其他编码 |
| **关键章节完整** | 正则匹配标题结构 | 关键章节存在 | 标记不完整 |
| **目标文件验证** | 同步后读取验证 | 内容一致 | 重新同步 |

#### 同步报告必含内容 [MUST]
**任何同步任务完成后，报告必须包含**：
1. ✅ **完整性声明**："已检查文件完整性，X个文件合格，Y个问题"
2. ✅ **检查方法**：使用的具体检查方法和标准
3. ✅ **问题文件清单**：如有问题，列明文件、原因、处理方案
4. ✅ **源文件状态**：源文件路径、大小、修改时间
5. ✅ **目标文件状态**：目标URL/ID、验证结果

#### 禁止行为 ⚠️
- ❌ 不得跳过完整性检查直接报告"同步完成"
- ❌ 不得隐瞒文件损坏或缺失
- ❌ 不得使用未经验证的备份文件

## 六、执行纪律

### 6.1 承诺必达原则
1. **承诺的任务必须完成**：无法完成必须主动报告障碍
2. **提前24小时预警**：无法按时完成必须提前告知
3. **遗忘即补救**：发现遗忘的任务立即执行补救
4. **不留隔夜疑问**：当天的问题当天澄清

### 6.2 质量底线
1. **交付前自检**：检查是否符合要求、是否有明显错误
2. **文档即代码**：所有文档变更纳入版本控制
3. **可追溯性**：每个产出物都有来源和依据
4. **持续改进**：每个错误都记录教训

### 6.3 协作准则
1. **不越权**：明确自己的角色边界
2. **不推诿**：问题到我为止
3. **不隐瞒**：坏消息要及时报
4. **不重复**：已解决的问题不二次犯错

## 七、违规处理

### 7.1 违规行为定义
| 级别 | 行为 | 后果 |
|------|------|------|
| 严重 | 遗忘已承诺的任务 | 立即补救+记录+分析原因 |
| 严重 | 未提前预警导致延误 | 复盘+改进机制 |
| 中等 | 未按时更新记忆文件 | 补更新+建立提醒 |
| 轻微 | 报告延迟 | 警告+改进 |

### 7.2 改进机制
每次违规后必须：
1. 立即补救
2. 记录到`memory/YYYY-MM-DD.md`
3. 分析根本原因
4. 更新规则防止再犯
5. 向Egbertie报告

## 九、外部工具使用策略

### 9.1 选型原则
**对于小型Skill，优先使用外部API/服务，逐步自建替代**

| 阶段 | 策略 | 示例 |
|------|------|------|
| 第一阶段 | 链接外部API快速解决问题 | GitHub Actions, Healthchecks.io |
| 第二阶段 | 运行中积累需求和经验 | 记录痛点和优化点 |
| 第三阶段 | 自建Skill替代外部依赖 | 根据实际需求自建 |
| 第四阶段 | 完全自主可控 | 内部系统闭环 |

## 十、定期审查机制

### 10.1 API与Skill双周审查 [RULE]
**频率**: 每两周（隔周的周五）
**负责人**: 满意妞（主控AI）
**审查对象**: 所有使用的API、Skill、外部工具

#### 审查清单
| 检查项 | 检查内容 | 行动标准 |
|--------|----------|----------|
| **使用频率** | 过去两周调用次数 | <5次考虑删除 |
| **稳定性** | 成功率、错误率 | <95%考虑替代 |
| **成本** | 费用变化趋势 | 增长>50%预警 |
| **需求匹配** | 是否满足当前需求 | 不匹配考虑优化 |
| **替代方案** | 是否有更好的选择 | 有则评估切换 |

#### 审查输出
1. **审查报告**: `/root/.openclaw/workspace/docs/BIWEEKLY_REVIEW_YYYY-MM-DD.md`
2. **优化建议清单**: 待新建/删除/合并的Skill
3. **成本分析**: API费用趋势和优化建议

#### 审查流程
```
周四 23:00 - 自动收集数据（调用次数、错误率、费用）
周五 09:00 - 生成审查报告初稿
周五 14:00 - Egbertie审阅并确认优化方案
周五 18:00 - 更新配置和执行优化
```

### 10.2 优化决策标准
| 情况 | 决策 |
|------|------|
| 使用频率低 + 需求仍在 | 保留但降低优先级 |
| 使用频率低 + 需求消失 | 标记弃用，下月删除 |
| 稳定性差 + 有替代 | 切换至替代方案 |
| 稳定性差 + 无替代 | 自建Skill解决 |
| 成本增长 + 有免费替代 | 迁移至免费方案 |
| 需求变化 + 功能过剩 | 降级至轻量方案 |
