# Skill 运营管理规则 V1.0

> **制定者**: 满意解研究所  
> **生效日期**: 2026-03-14  
> **适用范围**: 所有 OpenClaw Skill（外部+自建）  
> **核心原则**: 评估严谨、安全可控、使用规范、自建优先、及时更新

---

## 一、Skill 全生命周期管理

### 1.1 生命周期流程

```
发现/需求 → 评估 → 安装 → 使用 → 监控 → 更新/淘汰
    ↑___________________________________________↓
```

### 1.2 各阶段责任人

| 阶段 | 责任人 | 关键动作 |
|------|--------|----------|
| 发现/需求 | 全员 | 提出skill需求，说明业务价值 |
| 评估 | 满意妞 | 安全审计、成本评估、功能匹配 |
| 决策 | Egbertie | 最终决定是否安装 |
| 安装 | 满意妞 | 执行安装，记录到信赖清单 |
| 使用 | 各角色 | 按规范使用，反馈问题 |
| 监控 | 满意妞 | 监控成本、安全、使用频率 |
| 更新/淘汰 | 满意妞 | 定期更新，淘汰无用skill |

## 二、Skill 评估规则（安装前必做）

### 2.1 评估维度

| 维度 | 权重 | 评估内容 | 通过标准 |
|------|------|----------|----------|
| **安全** | 40% | 代码审计、权限检查、数据流向 | 无高危漏洞，无敏感权限滥用 |
| **成本** | 30% | API费用、运行成本、维护成本 | 符合预算，有免费替代优先 |
| **功能** | 20% | 功能匹配度、与现有skill重复度 | 功能必要，无明显重复 |
| **可维护** | 10% | 更新频率、社区活跃度、文档质量 | 有维护，文档完整 |

### 2.2 评估流程

**Step 1: 安全审计**（必做）
- [ ] 读取 SKILL.md 全文
- [ ] 检查 requires 字段（依赖哪些工具/权限）
- [ ] 检查是否有网络请求（curl/wget/外部API）
- [ ] 检查是否有文件写操作（特别是系统路径）
- [ ] 检查是否有执行权限（exec/subprocess）

**Step 2: 成本评估**（必做）
- [ ] 是否需要外部API key？
- [ ] API调用是否收费？免费额度多少？
- [ ] 是否有本地替代方案？
- [ ] 月度预估成本（按日均使用10次计算）

**Step 3: 功能评估**（必做）
- [ ] 与现有skill功能是否重复？
- [ ] 是否可以用现有skill组合实现？
- [ ] 业务价值评分（1-10分，低于6分不装）

**Step 4: 决策记录**（必做）
- 评估结果写入 `skill-evaluation-summary.md`
- 决策（安装/暂缓/拒绝）及理由
- 如拒绝，说明自建替代方案

### 2.3 评估文档模板

```markdown
## Skill评估: [skill名称]

**评估日期**: YYYY-MM-DD  
**评估人**: 满意妞

### 安全审计
- 网络请求: 有/无
- 敏感权限: 有/无（具体说明）
- 数据流向: 本地/外部（具体说明）
- 风险等级: 🟢低/🟡中/🔴高

### 成本评估
- 外部API: 需要/不需要
- 免费额度: XXX次/月
- 预估成本: ¥X/月
- 本地替代: 有/无

### 功能评估
- 与现有skill重复: 是/否（与XXX重复）
- 业务价值: X/10分
- 可替代性: 高/中/低

### 决策
- [ ] 安装（理由: XXX）
- [ ] 暂缓（理由: XXX，预计重评时间: XXX）
- [ ] 拒绝（理由: XXX，自建替代: XXX）
```

## 三、Skill 安全规则（使用中必遵）

### 3.1 绝对禁止（红线）

1. **禁止安装未审计skill**
   - 所有skill必须经过完整评估流程
   - 禁止直接从互联网下载安装未知来源skill

2. **禁止授予过度权限**
   - 只授予skill功能必需的最小权限
   - 定期检查已安装skill的权限范围

3. **禁止在生产环境测试新skill**
   - 新skill必须在隔离环境测试
   - 测试通过后方可正式使用

4. **禁止使用过旧版本skill**
   - 发现安全漏洞的版本立即停用
   - 超过6个月未更新的skill需重新评估

### 3.2 安全监控

**每日自动检查**:
- skill文件完整性（是否有未授权修改）
- skill调用日志（是否有异常行为）
- 外部网络请求（是否有未预期流量）

**每周人工审计**:
- 抽查skill输出质量
- 检查是否有敏感信息泄露
- 评估skill是否仍符合业务需求

## 四、Skill 使用规则

### 4.1 使用优先级

```
自建skill > 已审计外部skill > 新外部skill
```

**解释**:
- 优先使用自建skill（安全可控，成本最优）
- 其次使用已审计并长期稳定运行的外部skill
- 新外部skill仅在必要时使用，且需经过完整评估

### 4.2 成本控制

**模型使用优先级**（已在各skill中固化）:
1. **Kimi**（包月）→ 日常对话、中文处理、标准分析
2. **MiniMax/Moonshot**（便宜）→ 数据分析、内容生成
3. **GPT-4**（中等）→ 复杂推理、英文内容
4. **Claude**（贵）→ 高精尖代码、深度分析（需审批）

**调用频率控制**:
- 相同输入缓存复用（30天）
- 批量任务优先使用本地工具（Python/DuckDB）
- 非紧急任务延迟到夜间（API费率可能更低）

### 4.3 使用记录

**每次使用需记录**:
- 使用时间
- 使用skill名称
- 输入摘要（脱敏）
- 输出结果评价（有用/无用）
- 遇到的问题（如有）

**记录位置**: `memory/skill-usage-log.md`

## 五、Skill 自建替代原则

### 5.1 自建优先场景

以下情况**必须**自建替代:
1. 外部skill成本过高（月费>¥100）
2. 外部skill权限过度（需系统级权限）
3. 外部skill功能与现有体系冲突
4. 外部skill长期未更新（>6个月）
5. 外部skill涉及敏感数据处理

### 5.2 自建流程

**Step 1: 需求分析**
- 明确自建skill要解决的问题
- 列出必需功能和可选功能
- 评估开发成本（时间+技术难度）

**Step 2: 技术选型**
- 选择实现语言（Python/Bash/Node）
- 确定依赖工具（本地优先，减少外部依赖）
- 设计输入输出格式

**Step 3: 开发实现**
- 编写SKILL.md（必须包含安全声明）
- 实现核心功能（本地处理优先）
- 添加错误处理和日志记录

**Step 4: 测试验证**
- 功能测试（各种输入场景）
- 安全测试（权限边界检查）
- 成本测试（API调用次数统计）

**Step 5: 文档完善**
- 使用说明
- 示例对话
- 更新日志

### 5.3 自建skill模板

见 `skills/template/SKILL_TEMPLATE.md`

## 六、Skill 更新规则

### 6.1 更新触发条件

**必须立即更新**:
- 发现安全漏洞
- 依赖服务变更导致不可用
- 业务逻辑错误影响决策质量

**计划更新**（每月第一个周六）:
- 检查所有skill更新
- 评估新版本价值
- 更新必要skill

### 6.2 更新流程

**Step 1: 检查更新**
```bash
# 检查外部skill更新
clawhub list --outdated

# 检查自建skill是否需要迭代
# 基于使用反馈和功能需求
```

**Step 2: 评估更新价值**
- 更新内容是否涉及安全修复？
- 更新内容是否带来新功能？
- 更新是否引入破坏性变更？
- 更新成本（时间/学习成本）vs 收益

**Step 3: 备份旧版本**
```bash
cp -r skills/skill-name skills/skill-name-backup-$(date +%Y%m%d)
```

**Step 4: 执行更新**
```bash
# 外部skill
clawhub update skill-name

# 自建skill
# 手动更新文件，测试通过后替换
```

**Step 5: 验证测试**
- 核心功能是否正常？
- 与其他skill是否兼容？
- 成本是否符合预期？

**Step 6: 更新记录**
- 更新 SKILL.md 中的版本号和更新日志
- 更新 MEMORY.md 中的skill清单
- 记录到 `skill-update-log.md`

### 6.3 版本管理

**版本号规则**:
- 主版本号.次版本号.修订号（如 v1.2.3）
- 主版本号：重大功能变更，可能不兼容
- 次版本号：新功能，向下兼容
- 修订号：bug修复，安全补丁

**版本记录位置**:
- 每个skill的 `_meta.json` 中 `"version"` 字段
- `MEMORY.md` 中的skill清单
- `skill-update-log.md` 更新日志

## 七、Skill 淘汰机制

### 7.1 淘汰标准

出现以下情况之一，skill进入淘汰评估:
- 连续30天未被使用
- 功能被新skill完全替代
- 维护成本过高（月成本>¥200且无法降低）
- 存在无法修复的安全隐患
- 依赖服务停止维护

### 7.2 淘汰流程

**Step 1: 标记待淘汰**
- 在 `MEMORY.md` 中标记状态为 "⏳ 待淘汰"
- 记录淘汰原因

**Step 2: 寻找替代方案**
- 是否有其他skill可替代？
- 是否需要自建替代？
- 替代方案评估

**Step 3: 数据迁移**
- 导出该skill产生的历史数据
- 迁移到替代方案或归档

**Step 4: 执行淘汰**
```bash
# 备份
mv skills/skill-name skills/archive/skill-name-$(date +%Y%m%d)

# 或删除（确认无重要数据后）
rm -rf skills/skill-name
```

**Step 5: 更新记录**
- 更新 `MEMORY.md`
- 记录到 `skill-elimination-log.md`
- 通知所有使用者

## 八、工作空间完整性巡查机制

### 8.1 巡查目标

基于第一性原理，确保工作空间**结构完整、内容时效、引用一致、安全合规**。

**核心原则**:
- 文件夹是信息的"容器"，容器缺失则信息无处存放
- 文件内容的"过时"比"缺失"更隐蔽，危害更大
- 引用的"不一致"会导致决策基于错误信息

### 8.2 巡查维度（4维检查法）

| 维度 | 检查内容 | 频率 | 工具 |
|------|----------|------|------|
| **结构完整性** | 空文件夹、文件夹缺失、命名规范 | 每日09:00 | `workspace-integrity-guardian` |
| **内容时效性** | 日期过时、版本不一致、状态未更新 | 每日09:00 | `workspace-integrity-guardian` |
| **引用一致性** | 路径有效性、索引同步、链接检查 | 每周六 | `workspace-integrity-guardian` |
| **安全合规性** | 敏感文件权限、密码泄露、PII脱敏 | 每日+实时 | `workspace-integrity-guardian` |

### 8.3 执行流程

**自动巡查（每日09:00）**:
1. 扫描空文件夹
2. 检测过时日期引用
3. 验证关键路径有效性
4. 生成巡查报告

**人工复核（每周六10:00）**:
1. 查看自动巡查报告
2. 抽查关键文件内容
3. 验证引用路径
4. 更新全局目录索引

### 8.4 异常响应

**实时告警**（立即通知）:
- 敏感文件权限变更
- 密码/密钥泄露
- 核心文件夹被删除

**日报告警**（每日09:30）:
- 发现空文件夹
- 内容过时>3天
- 引用失效

**响应方式**:
- 主通道：Kimi Claw即时消息
- 第二通道：飞书（紧急情况）

## 九、Skill整合优化机制

### 9.1 优化目标

基于第一性原理，避免Skill**过多、重叠、碎片化**。

**目标状态**:
```
当前: 25个Skill（杂乱、重复、难管理）
  ↓ 整合优化
目标: 8-12个核心Skill（精简、统一、高效）
```

### 9.2 碎片化诊断标准

| 诊断项 | 标准 | 当前状态 | 优化目标 |
|--------|------|----------|----------|
| **数量过载** | >15个 | 25个 🔴 | <=12个 |
| **功能重叠** | 重叠度>30% | 平均45% 🔴 | <15% |
| **使用不均** | 80%使用来自20%Skill | 未统计 🟡 | 符合帕累托法则 |
| **管理成本** | 月管理时间>20h | 50h 🔴 | <20h |

### 9.3 整合优化策略

**策略1: 功能合并（Merge）**
- adwords + copywriting → `marketing-content-suite`
- automate-excel + csvtoexcel → `excel-processor`
- 效果: 减少2-4个Skill

**策略2: 自建替代（Replace）**
- agent-orchestrator → 自建 `task-coordinator`
- agents-manager → 自建 `role-manager`
- 效果: 提升可控性，降低外部依赖

**策略3: 延迟安装（Defer）**
- 非刚需Skill等真正需要时再装
- 高成本Skill等预算充足再装
- 效果: 避免资源浪费

**策略4: 直接淘汰（Eliminate）**
- 功能被完全替代的Skill
- 长期未使用（>90天）的Skill
- 效果: 精简体系，降低负担

### 9.4 整合执行计划

**第一阶段（本周）**:
- 合并 adwords + copywriting
- 合并 automate-excel + csvtoexcel
- 7个Skill → 5个套件

**第二阶段（本月）**:
- 新安装时直接安装整合版
- 新增9个 → 5个套件

**第三阶段（本季度）**:
- 建立持续整合优化机制
- 每月评估，季度整合，年度精简

### 9.5 持续优化机制

**每月评估**（第一个周六）:
- 检查可合并的Skill
- 评估使用频率分布
- 制定下月整合计划

**季度整合**（每季度末）:
- 执行大规模功能合并
- 体系化重构

**年度精简**（每年12月）:
- 淘汰长期未使用的Skill
- 清理被替代的Skill
