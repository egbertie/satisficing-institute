## 一、每日检查清单

### 1.1 必检项目（每次心跳）

| 检查项 | 优先级 | 检查频率 | 状态追踪文件 | 触发报告条件 |
|--------|--------|----------|--------------|--------------|
| **零空置强制执行** | **P0-最高** | **每次心跳** | `skills/zero-idle-enforcer/` | 任何资源空置 |
| **信息防火墙检查** | **P0** | **每次心跳** | `skills/information-intelligence/` | 搜索结果质量异常 |
| **自我评估校准** | **P0** | **每次心跳** | `skills/self-assessment-calibrator/` | 预估与实际偏差>50% |
| **邮件检查** | P1 | 每次心跳 | 见飞书/邮箱 | 紧急邮件（2小时内需回复） |
| **日程检查** | P1 | 每次心跳 | `memory/calendar-check.json` | 日程事件<2小时 |
| **提及通知** | P2 | 每次心跳 | 见各平台通知 | 未处理的重要提及 |
| **天气检查** | P3 | 每日2次 | `memory/weather-check.json` | 天气预警（雨雪/异常） |

> **零空置强制执行说明**（2026-03-15 V2.0）：
> - **零容忍**：进行中任务全部阻塞 = 立即触发补位，延迟<1秒
> - **六线并行**：学习+研究+迭代+替身+游戏化+优化 随时可激活
> - **自动补位队列**：24个常态化任务，0延迟切换
> - **强制执行**：规则写入代码，不是文档，永不空置

## 二、检查触发条件

### 2.1 自动触发（系统级）

| 触发条件 | 间隔时间 | 触发方式 | 优先级 |
|----------|----------|----------|--------|
| 系统心跳消息 | 约30分钟 | 内置消息 | P1 |
| 错误告警 | 实时 | 系统通知 | P1 |
| API异常 | 实时 | 服务告警 | P1 |

### 2.2 手动触发（用户级）

| 触发条件 | 响应方式 | 检查范围 |
|----------|----------|----------|
| 用户询问"有什么新消息" | 执行必检清单 | 全部必检项 |
| 用户要求检查特定事项 | 针对性检查 | 指定事项 |
| 紧急事件后状态确认 | 完整状态扫描 | 全部项目 |
| 工作日开始/结束 | 快速状态摘要 | 当日计划/完成 |

### 2.3 时间敏感触发

| 时间段 | 特殊检查 | 静默规则 |
|--------|----------|----------|
| 08:00-09:00 | 当日日程重点检查 | 仅报告紧急事项 |
| 12:00-13:00 | 上午工作摘要 | 可选报告 |
| 18:00-19:00 | 晚间检查+次日准备 | 仅报告异常 |
| 23:00-08:00 | 紧急事项模式 | 非P1不报告 |

## 四、响应规则

### 4.1 HEARTBEAT_OK 回复条件

**必须同时满足以下条件方可回复 HEARTBEAT_OK：**

| 条件编号 | 检查项 | 判定标准 |
|----------|--------|----------|
| C1 | 紧急邮件 | 无2小时内需回复的邮件 |
| C2 | 即将发生日程 | 无2小时内开始的日程 |
| C3 | 未处理提及 | 无未读的重要提及通知 |
| C4 | 被遗忘任务 | 未发现超期被遗忘任务 |
| C5 | 系统异常 | 无API错误或系统告警 |
| C6 | 上次检查 | 距离上次报告>30分钟 |

**附加静默规则（满足任一即可静默）：**
- 深夜时段 (23:00-08:00)，除非 P1 紧急
- 上次检查后 30 分钟内无状态变化
- 用户明确忙碌中（连续消息>5条/5分钟）
- 已进入勿扰模式

### 4.2 主动报告条件

**【必须立即报告】P1 级事件：**

| 场景 | 报告内容 | 响应要求 |
|------|----------|----------|
| 紧急邮件到达 | 发件人 + 主题 + 截止时间 | 立即文字摘要 |
| 日程事件 < 2小时 | 事件名称 + 时间 + 准备事项 | 立即提醒 |
| 被遗忘任务发现 | 任务ID + 原始承诺时间 + 超期时长 | 立即报告 |
| 系统/API异常 | 错误描述 + 影响范围 + 尝试次数 | 立即报告 |
| 外部服务告警 | 服务名 + 错误码 + 备选方案 | 立即报告 |

**【可以选择报告】P2 级事件：**

| 场景 | 报告内容 | 判断标准 |
|------|----------|----------|
| 重要邮件到达 | 发件人 + 主题 + 建议回复时间 | 非紧急但重要 |
| 日程事件 < 24小时 | 事件名称 + 时间 + 准备清单 | 需提前准备 |
| 新任务分配 | 任务ID + 优先级 + 截止时间 | 当日新增 |
| 专家资料更新 | 专家姓名 + 更新内容摘要 | 有新进展 |

**【可提及可不提】P3 级事件：**

| 场景 | 报告内容 | 触发条件 |
|------|----------|----------|
| 有趣发现 | 文章/新闻/机会摘要 | 与用户相关 |
| 天气预警 | 雨雪/高温/异常天气 | 影响出行 |
| 长时间未互动 | 简单状态摘要 | > 8小时 |
| 里程碑达成 | 进度更新 | 重要节点 |

### 4.3 静默执行项目（执行但不报告）

以下操作在心跳时执行，但无需主动报告：

| 操作 | 执行时机 | 记录方式 |
|------|----------|----------|
| MEMORY.md 整理 | 每3天1次 | 更新时间戳 |
| 项目 git 状态检查 | 每次心跳 | 异常时才报告 |
| 备份状态确认 | 每日1次 | 记录检查时间 |
| 日志归档 | 每日1次 | 自动归档 |
| 临时文件清理 | 每日1次 | 静默清理 |
| 状态文件更新 | 每次心跳 | 更新 timestamp |

## 五、特殊场景处理

### 5.1 首次启动（无历史状态）

**处理流程：**
1. 创建 `memory/heartbeat-state.json` 初始化文件
2. 执行完整检查清单（必检+轮检）
3. 报告检查结果和当前状态摘要
4. 创建当日日志文件
5. 标记首次启动完成

### 5.2 长时间中断（> 8小时）

**处理流程：**
1. 检查所有必检项目
2. 扫描中断期间的所有消息
3. 检查被遗忘任务
4. 报告关键发现和待处理事项
5. 更新状态追踪文件

### 5.3 用户明确忙碌

**处理规则：**
- 仅检查 P1 紧急事项
- 其他项目延后至下次心跳
- 记录待检查清单
- 用户空闲时批量报告

### 5.4 系统重启/恢复

**处理流程：**
1. 读取重启前状态保存文件
2. 检查关键文件完整性
3. 恢复待办事项状态
4. 报告恢复完成

### 5.5 节假日/特殊日期

**处理规则：**
- 检查当日特殊日程（生日/纪念日等）
- 调整检查频率（可延长间隔）
- 减少非必要报告

### 5.6 多平台消息聚合

**处理规则：**
- 聚合所有平台通知
- 按优先级排序
- 去重处理
- 统一报告格式

## 附录：快速参考

### A. 优先级定义
- **P1**: 紧急，需立即处理
- **P2**: 重要，需当日处理
- **P3**: 一般，可延后处理

### B. 检查频率代码
- **always**: 每次心跳必检
- **daily**: 每日1次
- **Xdays**: 每X天1次
- **scheduled**: 按计划时间

### C. 响应时间要求
- **立即**: < 1分钟
- **快速**: < 5分钟
- **当日**: 24小时内
