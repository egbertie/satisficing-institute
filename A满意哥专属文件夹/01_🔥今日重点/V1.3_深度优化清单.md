## 二、搜索工具矩阵激活（Token降本80%）

### 2.1 统一搜索套件 (unified-search)
**核心能力**：
- 智能路由：根据查询类型自动选择最佳引擎
- 多引擎聚合：并行查询，结果去重
- 本地缓存：DuckDB缓存，24小时内直接返回
- 完全自建：替代4个外部搜索Skill

**应用场景**：
```
硬科技投资趋势 → 自动路由到Tavily+Brave聚合
中文政策解读 → 路由到百度+Bing CN
学术论文查找 → 路由到Exa+Google Scholar
深度网页内容 → 路由到Firecrawl
```

### 2.2 多搜索引擎 (multi-search-engine)
**17引擎矩阵**：
- 中文8个：百度、Bing CN、360、搜狗、微信、头条、集思录
- 国际9个：Google、DuckDuckGo、Yahoo、Startpage、Brave、Ecosia、Qwant、WolframAlpha

**应用场景**：
- 政策信息：百度+微信文章+头条
- 国际市场：Google+Brave+Startpage
- 计算查询：WolframAlpha（汇率、数学、天气）
- 隐私保护：DuckDuckGo+Qwant

### 2.3 学术深度研究 (academic-deep-research)
**四层验证体系**：
1. 真实性验证：机构/数据/案例/专家验证
2. 引用标准化：^ID^学术引用微格式
3. 红军审查：方法论/证据/逻辑/立场攻击
4. 蓝军防御：承认局限/补充论证/稳健性检验

**应用场景**：
- 所有对外发布的报告必须通过四层验证
- 数据标注：✓已验证 / ⚠️待验证 / 🔶模拟数据 / 🔷模拟案例

### 2.4 智能网页抓取 (smart-web-fetch)
**四级降级策略**：
1. Jina Reader（首选，免费，中文好）
2. markdown.new
3. defuddle.md
4. 原始HTML兜底

**Token节省50-80%**：去除广告、导航栏等噪音

### 2.5 满意解Web抓取器 (satisficing-web-fetcher)
**Scrapling自建方案**：
- 反爬绕过
- 自适应解析
- 沙箱隔离
- 安全审计
- 零外部API成本

### 2.6 公司搜索 (company-search-kimi)
**应用场景**：
- 客户背景调研
- 竞品分析
- 投资标的研究

## 三、V1.3核心优化点

### 3.1 数据严谨性全面升级

**现状问题**：
- 部分数据标注为🔶[模拟数据]
- 引用格式不统一
- 缺乏红军/蓝军审查

**V1.3优化**：
- 所有数据必须通过academic-deep-research四层验证
- 建立标准化引用体系^ID^
- 主动暴露局限性，增强可信度

**具体执行**：
```
每条数据 → 多源搜索验证 → 标注来源 → 红蓝军审查 → 最终输出
```

### 3.2 搜索策略矩阵化

**查询分类与引擎匹配**：

| 查询类型 | 主引擎 | 备用引擎 | 验证方式 |
|---------|--------|---------|---------|
| 中文新闻/政策 | 百度+Bing CN | 搜狗+微信 | 多源交叉 |
| 中文百科/知识 | 百度 | 头条+集思录 | 事实核查 |
| 国际新闻 | Google+Brave | DuckDuckGo | 多源交叉 |
| 学术论文 | Exa+Google Scholar | 知网+万方 | 引用验证 |
| 技术文档 | Firecrawl | smart-web-fetch | 原文比对 |
| 公司信息 | company-search-kimi | 天眼查+企查查 | 数据交叉 |
| 计算/转换 | WolframAlpha | - | 直接计算 |

### 3.3 内容生产流程重构

**旧流程**：
```
需求 → Kimi生成 → 简单修改 → 输出
```

**V1.3新流程**：
```
需求 → 多源搜索验证 → Kimi生成 → academic-deep-research四层验证 
→ 红军/蓝军审查 → 引用标准化 → 真实性标注 → 最终输出
```

**预期效果**：
- 数据准确率提升90%+
- Token消耗降低80%（缓存+智能路由）
- 学术严谨性达到可发表水平
