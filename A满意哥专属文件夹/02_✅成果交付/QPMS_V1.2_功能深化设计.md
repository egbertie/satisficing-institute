## 一、版本概述

### 1.1 版本演进
QPMS从V1.1的核心评估框架向V1.2的深度诊断系统跃迁，标志着产品从"感知层"向"干预层"的战略升级。

**V1.1核心能力回顾**：
- 六维评估可视化：合伙人基础画像
- 感知力雷达图：直觉与数据的融合呈现
- PFI指数（Partnership Fit Index）：匹配度量化
- 风险热力图：风险分布态势感知

**V1.2新增定位**：  
基于4个典型失败案例（正浩创新股权稀释、配送机器人CTO失控、纳微科技路线分歧、寒武纪团队冲突），构建合伙人关系的"预警-诊断-干预"闭环。

## 三、新增模块详细规格

### 3.1 模块一：股权设计诊断模块 (Equity Design Diagnostic Module)

#### 3.1.1 输入数据

| 数据类型 | 具体字段 | 数据来源 |
|---------|---------|---------|
| **股权结构** | 各股东持股比例、持股类型（直接/间接/代持）、股权获得时间 | 股东协议、公司章程 |
| **Vesting条款** | 归属期总时长、已归属比例、归属触发条件、加速条款 | 股权激励计划书 |
| **里程碑设定** | 技术/商业里程碑定义、达成标准、未达成惩罚机制 | 投资协议、对赌条款 |
| **稀释历史** | 融资轮次、估值变化、股权稀释比例、期权池预留 | 融资文档、Cap Table |
| **代持/期权** | 代持协议内容、期权池规模、授予对象清单 | 代持协议、期权计划 |

#### 3.1.2 处理逻辑

**算法1：Vesting健康度评分**
```
Vesting_Score = (基础归属合理性 × 0.3) + (里程碑对齐度 × 0.3) + (条款完整性 × 0.2) + (退出保护机制 × 0.2)

• 基础归属合理性：归属期时长（行业标准4年为佳）
• 里程碑对齐度：技术里程碑与商业里程碑的匹配程度
• 条款完整性：是否包含 cliffs、加速归属、回购条款
• 退出保护机制：离职时的股权处理规则明确性
```

**算法2：稀释风险预警模型**
```
Dilution_Risk = (当前持股比例) × (预计融资轮次) × (行业稀释系数) / (反稀释保护强度)

预警等级：
- 🔴 高风险：预计上市后创始团队持股 < 15%
- 🟡 中风险：预计上市后创始团队持股 15%-30%
- 🟢 低风险：预计上市后创始团队持股 > 30%
```

**算法3：代持/期权风险识别**
```
检查项：
1. 代持协议是否经过公证
2. 期权池是否预留充足（建议10-15%）
3. 核心技术人员是否有足额期权绑定
4. 代持人是否存在债务/婚姻风险
```

#### 3.1.3 输出结果

| 输出项 | 内容描述 | 格式 |
|-------|---------|------|
| **股权健康度评分** | 0-100分综合评分，含各子项得分 | 数字 + 雷达图 |
| **稀释路径模拟** | 未来3轮+IPO的股权稀释预测 | 瀑布图 + 表格 |
| **风险点清单** | 按严重程度排序的股权风险清单 | 优先级列表 |
| **优化建议** | 针对发现问题的具体改进方案 | 结构化建议书 |

#### 3.1.4 可视化方式

- **股权结构树图**：展示多层股权结构，标注代持关系
- **Vesting进度条**：可视化归属进度与关键时间节点
- **稀释瀑布图**：融资轮次对股权的逐级影响
- **风险热力矩阵**：横轴为影响程度，纵轴为发生概率

### 3.2 模块二：关键人风险评估模块 (Key Person Risk Assessment Module)

#### 3.2.1 输入数据

| 数据类型 | 具体字段 | 数据来源 |
|---------|---------|---------|
| **人员信息** | 核心团队成员名单、职位、入职时间、持股比例 | HR系统、组织架构图 |
| **技术依赖** | 代码贡献统计、专利发明人、技术文档作者 | Git仓库、专利数据库 |
| **文档清单** | 技术文档完备性、知识转移文档、SOP手册 | 文档管理系统 |
| **竞业条款** | 竞业禁止协议、保密协议、知识产权归属协议 | 劳动合同附件 |
| **离职影响** | 过往离职案例、离职交接情况、客户/供应商关系 | 人事档案 |

#### 3.2.2 处理逻辑

**算法1：技术依赖度评估**
```
Tech_Dependency_Index = (代码贡献占比 × 0.4) + (专利依赖度 × 0.3) + (文档完备度反向 × 0.2) + (替代难度 × 0.1)

评估维度：
• 代码贡献占比：核心模块代码量 / 总代码量
• 专利依赖度：核心技术专利第一发明人占比
• 文档完备度：API文档、架构文档、部署手册的完整程度
• 替代难度：该岗位市场人才供给稀缺度
```

**算法2：文档完备性检查**
```
Doc_Completeness = Σ(关键文档类型得分 × 权重)

检查清单（共10类文档）：
1. 系统架构设计文档
2. 核心算法说明文档
3. API接口文档
4. 数据库设计文档
5. 部署运维手册
6. 故障排查指南
7. 代码规范文档
8. 技术决策记录（ADR）
9. 第三方依赖清单
10. 技术债务清单
```

**算法3：离职影响模拟**
```
Impact_Score = (业务中断风险 × 0.35) + (知识流失风险 × 0.3) + (团队士气影响 × 0.2) + (外部关系损失 × 0.15)

模拟场景：
• 立即离职（无交接）：影响最大化
• 30天标准交接：影响中等
• 90天延长交接：影响最小化
• 渐进式过渡（兼职顾问）：平滑过渡
```

#### 3.2.3 输出结果

| 输出项 | 内容描述 | 格式 |
|-------|---------|------|
| **关键人风险矩阵** | 按人员维度的多维度风险评估 | 气泡图/矩阵图 |
| **技术依赖热力图** | 核心系统 vs 核心人员依赖关系 | 网络关系图 |
| **文档完备度评分** | 各模块文档完整度评分 | 进度条矩阵 |
| **离职情景分析报告** | 不同离职场景的模拟影响 | 情景对比表 |

#### 3.2.4 可视化方式

- **关键人风险矩阵**：横轴为不可替代性，纵轴为离职概率，气泡大小为影响范围
- **知识依赖网络图**：节点为人员与知识模块，边为依赖关系
- **文档完备度仪表板**：各领域文档覆盖率的进度条展示
- **离职影响对比图**：不同交接方案的影响程度对比

### 3.3 模块三：价值观对齐评估模块 (Values Alignment Assessment Module)

#### 3.3.1 输入数据

| 数据类型 | 具体字段 | 数据来源 |
|---------|---------|---------|
| **测评问卷** | 技术vs商业倾向、风险偏好、时间取向、伦理底线 | QPMS内置问卷 |
| **决策历史** | 过往重大决策记录、决策理由、执行结果 | 会议记录、决策文档 |
| **公开言论** | 媒体采访、演讲内容、社交媒体观点 | 公开信息抓取 |
| **行为模式** | 资金使用偏好、人才招聘标准、合作方选择 | 财务/HR数据分析 |
| **冲突记录** | 历史分歧事件、处理方式、最终结果 | 访谈与问卷 |

#### 3.3.2 处理逻辑

**算法1：四维价值观画像**
```
Values_Profile = {
    Tech_Business_Axis: [-10, +10],    // -10纯技术导向，+10纯商业导向
    Risk_Preference: [-10, +10],       // -10极度保守，+10激进冒险
    Time_Orientation: [-10, +10],      // -10长期主义，+10短期变现
    Ethical_Baseline: [0, 10]          // 0无底线，10极高伦理标准
}

维度解读：
• Tech-Business Axis：对技术理想vs商业现实的偏好
• Risk Preference：对不确定性的容忍度
• Time Orientation：对短期回报vs长期价值的权衡
• Ethical Baseline：商业伦理底线的高度
```

**算法2：价值观冲突预警**
```
Alignment_Score = 100 - Σ|Partner_A_Axis_i - Partner_B_Axis_i| × Weight_i

冲突阈值：
• ⚠️ 轻微差异：单项差异 < 3分
• 🔶 显著分歧：单项差异 3-6分
• 🚨 严重冲突：单项差异 > 6分

典型冲突组合：
• 技术极客 vs 商业操盘手：产品路线分歧
• 风险厌恶 vs 激进扩张：融资节奏冲突
• 长期主义 vs 短期变现：退出时机争议
```

**算法3：决策风格分析**
```
Decision_Style = (数据依赖度 × 0.25) + (直觉依赖度 × 0.25) + (协商倾向 × 0.25) + (决断速度 × 0.25)

分析维度：
• 数据驱动 vs 直觉驱动
• 独立决策 vs 集体协商
• 快速决断 vs 深思熟虑
• 结果导向 vs 过程合规
```

#### 3.3.3 输出结果

| 输出项 | 内容描述 | 格式 |
|-------|---------|------|
| **合伙人价值观雷达图** | 四维价值观的合伙人对比 | 多边形雷达图 |
| **价值观冲突矩阵** | 各合伙人组合的价值观差异热力图 | 热力矩阵 |
| **决策风格匹配度** | 决策方式的互补性与冲突分析 | 对比分析图 |
| **潜在冲突预警** | 基于价值观差异的冲突场景预测 | 风险列表 |

#### 3.3.4 可视化方式

- **价值观雷达图**：四维轴上的多合伙人对比
- **冲突热力图**：合伙人两两组合的价值观差异度
- **决策流程图**：不同决策风格下的流程模拟
- **情景冲突卡**：典型冲突场景的可视化呈现

### 3.4 模块四：战略分歧预警模块 (Strategic Divergence Warning Module)

#### 3.4.1 输入数据

| 数据类型 | 具体字段 | 数据来源 |
|---------|---------|---------|
| **战略文档** | 商业计划书、技术路线图、市场规划 | 公司内部文档 |
| **会议记录** | 战略讨论会议纪要、分歧点记录 | 会议系统 |
| **资源分配** | 研发投入占比、市场投入占比、人员配置 | 财务/HR系统 |
| **市场信号** | 竞品动态、行业趋势、技术演进 | 外部数据抓取 |
| **历史决策** | 过往战略调整记录、调整原因 | 决策档案 |

#### 3.4.2 处理逻辑

**算法1：技术路线分歧识别**
```
Tech_Divergence_Index = (技术路径差异度 × 0.4) + (研发投入分散度 × 0.3) + (专利布局冲突 × 0.3)

识别信号：
• 同一技术问题存在多个并行方案
• 研发团队被拆分到不同技术路线
• 关键技术决策长期处于悬而未决状态
• 技术负责人与商业负责人对优先级排序不一致
```

**算法2：市场时机判断差异**
```
Timing_Divergence = |Partner_A_Timing_Score - Partner_B_Timing_Score|

时机评估维度：
• 产品上市时机（早/恰当时/晚）
• 市场扩张节奏（激进/稳健/保守）
• 融资窗口判断（立即/等待更佳时机）
• 竞争应对策略（主动进攻/防守观望）
```

**算法3：多元化倾向评估**
```
Diversification_Risk = (业务线数量 × 资源分散系数) / (核心能力聚焦度)

风险信号：
• 同时推进超过3条非相关业务线
• 核心业务收入占比持续下降
• 团队扩张方向与核心能力边界不匹配
• 资源分配呈现"撒胡椒面"特征
```

**算法4：战略一致性评分**
```
Strategic_Alignment = (愿景一致性 × 0.25) + (路径共识度 × 0.25) + (节奏协调性 × 0.25) + (资源配置合理性 × 0.25)

评分等级：
• 90-100：高度一致，执行顺畅
• 70-89：基本一致，微调即可
• 50-69：存在分歧，需要沟通
• <50：严重分歧，需要干预
```

#### 3.4.3 输出结果

| 输出项 | 内容描述 | 格式 |
|-------|---------|------|
| **战略一致性评分** | 整体战略对齐程度的量化评分 | 仪表盘 |
| **分歧热力图** | 各战略维度的合伙人观点分布 | 热力图 |
| **预警信号看板** | 需要关注的分歧信号清单 | 预警列表 |
| **情景模拟报告** | 不同战略选择的可能结果 | 决策树图 |

#### 3.4.4 可视化方式

- **战略分歧地图**：技术路线、市场时机、多元化三个维度的分歧可视化
- **资源流向图**：研发投入与战略方向的匹配度
- **决策时间轴**：关键战略决策的时间线与分歧点标注
- **情景对比卡**：不同战略选择的可能结果对比

### 5.2 干预建议框架

**Level 1: 信息干预**  
当风险评分在50-70区间，提供信息支持，帮助合伙人充分了解风险。

**Level 2: 流程干预**  
当风险评分在30-50区间，建议引入外部顾问或调整决策流程。

**Level 3: 结构干预**  
当风险评分<30，建议进行股权结构调整或团队重组。
