这句话如当头棒喝。

**顿悟**：
- 银行风控追求"最优解"，但真实世界没有最优解
- 创业需要的是"满意解"——在信息不全时做出有据可依的决策
- 合伙人选择这道生死题，没有教科书，没有标准答案

##### 2.3 方法论的构建

Egbertie开始系统整合：
- **金融风控的严谨性**——银行训练的底层逻辑

- **六祖慧能的顿悟思想**——东方智慧的压力转化
- **郭航江开源技术架构**——V1.4新增，系统鲁棒性保障

**这不是卖一套理论，是分享22年沉淀的真实阅历**。

- ✅ 所有专家都必须建立数字替身
- ✅ 体系内按数字替身模式运作

#### 4.4 专家报酬机制

**原则**：收入分成模式

1. **前期**：共同探讨，无预付费用
2. **有实际收费后**：按实际参与项目比例收费
3. **分配比例**：专家在其自身项目里占大头（确保有收获和参与动力）
4. **长期绑定**：优秀专家可转化为合伙人，享受长期收益分成

#### 6.5 QPMS量子感知引擎V2.0（ReACT模式）

**ReACT循环**：

```
Reasoning（推理）→ Acting（行动/调研）→ Observation（观察）→ Reasoning（推理）→ ...
```

**实施流程**：

| 轮次 | 阶段 | 动作 | 输出 |
|------|------|------|------|
| 1 | 初始推理 | 基于已有信息生成初步假设 | 假设清单 |
| 2 | 主动调研 | 针对假设缺口进行补充搜索 | 调研数据 |
| 3 | 修正推理 | 整合新信息，修正假设 | 修正报告 |
| 4 | 深度调研 | 针对关键疑点深入验证 | 验证证据 |
| 5 | 最终推理 | 生成最终结论与建议 | 完整报告 |

**证据链可追溯**：
- 每个结论都有明确的证据编号
- 支持"为什么这样说"的溯源查询
- 不确定结论明确标注置信度

**核心模块**：
1. **ECBM（认知偏见识别与缓解）**：9大认知偏见评估
2. **REPA（伦理对齐评估）**：价值观一致性检验
3. **ETDS（涌现匹配算法）**：动态匹配度计算

**9大认知偏见覆盖**：
- 确认偏误、沉没成本谬误、过度自信偏差
- 锚定效应、可得性启发、框架效应
- 群体思维、计划谬误、损失厌恶

**证据分级系统**：
- [V] 验证：经Web检索验证的学术来源
- [S] 合成：基于理论演绎的构建性内容
- [U] 待验证：原文声称但无法检索的特定数据

**四级降级链**：

```
┌─────────────────────────────────────────────────────────────┐
│  Level 1: 主服务 try                                        │
│  ├─ 正常调用主引擎                                          │
│  └─ 失败 → 降级到 Level 2                                   │
├─────────────────────────────────────────────────────────────┤
│  Level 2: 备用引擎 fallback                                 │
│  ├─ 切换到备用LLM/API                                       │
│  └─ 失败 → 降级到 Level 3                                   │
├─────────────────────────────────────────────────────────────┤
│  Level 3: 本地缓存 cache                                    │
│  ├─ 返回缓存结果（可能非最新）                               │
│  └─ 失败 → 降级到 Level 4                                   │
├─────────────────────────────────────────────────────────────┤
│  Level 4: 空结果兜底 empty                                  │
│  └─ 返回预设兜底响应，保证系统不崩溃                         │
└─────────────────────────────────────────────────────────────┘
```

**各工具降级策略**：

### 附录17: QPMS报告生成V2.0——ReACT模式 🔄

**借鉴来源**: 郭航江MiroFish项目ReportAgent

#### 17.1 升级背景

V1.0 QPMS报告生成采用"一次性输出"模式，虽然结构完整，但存在以下局限：
- 缺乏主动调研能力，依赖静态输入
- 推理过程不透明，难以追溯
- 复杂评估难以一次到位

V1.4引入**ReACT（Reasoning & Acting）模式**，实现多轮迭代、主动调研、证据链可追溯的报告生成。

#### 17.2 ReACT循环原理

ReACT = Reasoning（推理）+ Acting（行动/调研）+ Observation（观察）

```
┌─────────────────────────────────────────────────────────────┐
│                      ReACT 循环                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   ┌──────────┐      ┌──────────┐      ┌──────────┐        │
│   │ Reasoning│ ───→ │  Acting  │ ───→ │Observation│        │
│   │  (推理)   │      │ (行动)   │      │  (观察)   │        │
│   └──────────┘      └──────────┘      └─────┬────┘        │
│        ↑                                    │              │
│        └────────────────────────────────────┘              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 17.3 实施流程

**初始轮次 - 假设生成**：

| 步骤 | 动作 | 输出示例 |
|------|------|----------|
| Reasoning | 基于输入数据，生成初步假设 | "假设1：该创始人存在确认偏误" |
|  | 识别信息缺口 | "需要验证：创始人过往决策模式" |

**调研轮次 - 证据收集**：

| 步骤 | 动作 | 输出示例 |
|------|------|----------|
| Acting | 主动搜索相关信息 | 搜索创始人公开访谈、行业报道 |
| Observation | 记录调研发现 | "发现3次类似决策模式，2次成功1次失败" |

**修正轮次 - 假设更新**：

| 步骤 | 动作 | 输出示例 |
|------|------|----------|
| Reasoning | 整合新信息，修正假设 | "修正假设1：确认偏误存在但有限" |
|  | 识别新的信息缺口 | "需要验证：失败案例的具体原因" |

**深度轮次 - 精细验证**：

| 步骤 | 动作 | 输出示例 |
|------|------|----------|
| Acting | 针对性深入调研 | 搜索失败案例详情、行业分析 |
| Observation | 记录关键证据 | "失败主因是市场时机而非决策本身" |

**终轮 - 结论生成**：

| 步骤 | 动作 | 输出示例 |
|------|------|----------|
| Reasoning | 综合所有证据，生成最终结论 | "该创始人确认偏误程度：中等" |
|  | 明确置信度 | "置信度：75%，主要不确定性来自..." |

#### 17.4 证据链可追溯设计

**证据编号系统**：

```
EVIDENCE-[轮次]-[序号]-[类型]

示例：
- EVIDENCE-R1-001-WEB：第1轮调研，第1条证据，网页来源
- EVIDENCE-R2-003-INT：第2轮调研，第3条证据，访谈记录
```

**结论溯源示例**：

> **结论**：该创始人呈现中度确认偏误倾向
> 
> **证据链**：
> - 主证据：EVIDENCE-R1-002-WEB ^[1]^——创始人在3次访谈中均强调同一技术路线
> - 支持证据：EVIDENCE-R2-005-INT ^[2]^——团队成员反馈"较难说服其考虑替代方案"
> - 反证：EVIDENCE-R3-001-WEB ^[3]^——曾在2023年Q2调整过一次技术方向
> 
> **置信度**：70%（存在团队反馈样本量有限的局限）

**溯源查询接口**：
```python
def trace_conclusion(conclusion_id):
    """追溯结论的证据来源"""
    conclusion = get_conclusion(conclusion_id)
    evidence_chain = []
    for evidence_id in conclusion.evidence_ids:
        evidence = get_evidence(evidence_id)
        evidence_chain.append({
            'id': evidence_id,
            'source': evidence.source,
            'content': evidence.summary,
            'reliability': evidence.reliability
        })
    return evidence_chain
```

#### 17.5 主动调研能力

#### 18.4 降级决策逻辑

```python
class DegradationManager:
    def __init__(self):
        self.retry_count = 0
        self.max_retries = 2
        self.current_level = 1
    
    async def execute_with_degradation(self, task):
        """执行带降级的任务"""
        for level in range(self.current_level, 5):
            try:
                result = await self._execute_at_level(task, level)
                if result.success:
                    self.current_level = level  # 记录当前等级
                    return result
            except Exception as e:
                log.warning(f"Level {level} failed: {e}")
                continue
        
        # 全部失败，返回兜底
        return self._empty_fallback()
    
    def _execute_at_level(self, task, level):
        """在指定等级执行"""
        strategy = self._get_strategy(task.type, level)
        return strategy.execute(task)
```

#### 19.4 智能路由算法

```python
class LLMRouter:
    def __init__(self):
        self.creator = KimiK2P5()
        self.executor = DeepSeek()
        self.multimodal = Gemini()
        self.long_context = Kimi128K()
    
    def route(self, task):
        """智能路由决策"""
        
        # 1. 任务复杂度分析
        complexity = self._analyze_complexity(task)
        
        # 2. 特殊能力需求检测
        needs_multimodal = task.has_images or task.has_videos
        needs_long_context = task.token_count > 32000
        needs_creativity = task.requires_original_thinking
        
        # 3. 路由决策
        if needs_multimodal:
            return self.multimodal
        
        if needs_long_context:
            return self.long_context
        
        if complexity == "high" and needs_creativity:
            return self.creator
        
        # 默认使用执行者层
        return self.executor
    
    def _analyze_complexity(self, task):
        """分析任务复杂度"""
        factors = {
            'reasoning_steps': task.expected_reasoning_depth,
            'domain_knowledge': task.domain_specificity,
            'output_length': task.expected_output_length,
            'creativity_required': task.creativity_score
        }
        
        score = sum(factors.values())
        
        if score > 7:
            return "high"
        elif score > 4:
            return "medium"
        else:
            return "low"
```
