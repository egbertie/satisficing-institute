## 四、根因分析

### 4.1 为什么会信息不闭环

```
根因树:
├─ 流程问题
│  ├─ 有安排无确认（安排任务但不确认执行）
│  ├─ 有执行无汇报（做了但不告诉你）
│  └─ 有问题无升级（遇到困难不报告）
│
├─ 机制问题
│  ├─ 无闭环检查清单（不知道要汇报什么）
│  ├─ 无强制汇报触发（不汇报也没事）
│  └─ 无验收确认（你不确认就算完成）
│
└─ 文化问题
   ├─ 报喜不报忧（只说好听的）
   ├─ 被动响应（你不问我不说）
   └─ 追求数字（完成度好看就行）
```

### 4.2 系统性缺陷

| 缺陷 | 表现 | 后果 |
|------|------|------|
| 单向信息流动 | 我给你安排，但不反馈 | 你不知道进展 |
| 缺乏确认机制 | 做完了不请你验收 | 质量无法保证 |
| 异常隐藏 | 遇到问题不升级 | 问题扩大 |
| 进度虚报 | 数字好看实质不足 | 信任危机 |

## 五、信息闭环机制（立即建立）

### 5.1 闭环三原则

```
原则1: 安排必确认
  我安排任务 → 你确认收到 → 明确验收标准 → 约定汇报时间

原则2: 执行必汇报  
  开始执行 → 进度汇报 → 完成汇报 → 请你验收

原则3: 问题必升级
  遇到困难 → 立即报告 → 协商方案 → 确认后执行
```

### 5.2 闭环检查清单

**任务启动时（必须）**:
- [ ] 明确任务目标（SMART）
- [ ] 明确验收标准（量化）
- [ ] 明确完成时间（截止时间）
- [ ] 明确汇报节点（何时汇报）
- [ ] 获得你的确认（你同意才执行）

**任务执行中（必须）**:
- [ ] 按计划节点汇报
- [ ] 遇到阻塞立即报告
- [ ] 变更计划提前协商
- [ ] 记录执行日志

**任务完成时（必须）**:
- [ ] 交付物清单
- [ ] 验证方式说明
- [ ] 请你验收确认
- [ ] 记录完成时间

### 5.3 汇报模板

**启动汇报**:
```
【任务启动】XXX
【目标】具体要做什么
【标准】怎样算完成（量化）
【时间】预计何时完成
【节点】何时汇报进度
等待你的确认...
```

**进度汇报**:
```
【进度更新】XXX
【已完成】具体做了什么
【进行中】正在做什么
【阻塞】遇到什么问题（如有）
【变更】是否需要调整（如需）
```

**完成汇报**:
```
【任务完成】XXX
【交付物】清单+路径
【验证】怎么验证
【质量】自检结果
请你验收...
```
