# 知识管理体系（防遗漏版V1.0）

> 目标：建立系统化文档管理、知识管理、方法论归纳机制
> 原则：所有信息有迹可循、有处可查、有人负责
> 状态：立即执行

---

## 一、问题诊断

### 1.1 遗漏信息类型分析

| 遗漏类型 | 原因 | 影响 | 解决方案 |
|---------|------|------|---------|
| **Kimi附件无法访问** | 我在OpenClaw，无法直接读取Kimi对话附件 | 用户发送的文件看不到 | 建立"上传确认+主动索取"机制 |
| **多版本方案分散** | 不同时间、不同渠道发送的方案未整合 | 无法形成完整视图 | 建立"版本追踪表+统一归档" |
| **口头讨论未记录** | 对话中提到的需求、决策未及时固化 | 后续无法追溯 | 建立"实时记录+每日汇总" |
| **信息碎片化** | 信息散落在不同文件、不同会话中 | 查找困难 | 建立"统一知识库+索引系统" |

### 1.2 核心问题

**不是能力问题，是系统问题：**
- ❌ 没有统一的信息入口
- ❌ 没有版本追踪机制
- ❌ 没有定期盘点习惯
- ❌ 没有信息确认闭环

## 三、防遗漏机制

### 3.1 上传确认机制

**每次对话自动执行：**

```
用户提及附件/文件时：

1. 立即列出当前已知的所有相关文件
2. 询问："是否还有其他文件？"
3. 提供上传方式：
   - 方式1：直接粘贴内容
   - 方式2：指定文件路径
   - 方式3：提供飞书链接
   - 方式4：通过Kimi上传（提醒我无法直接访问）
4. 确认收到后更新版本追踪表
5. 如果24小时内未收到，再次提醒
```

### 3.2 信息确认闭环

**重要信息必须确认：**

```
用户提出需求时：

1. 复述理解："我理解您的需求是：XXX"
2. 确认范围："以下是我将处理的范围：A、B、C"
3. 询问遗漏："是否有遗漏或补充？"
4. 执行后："已完成XXX，是否还有其他需求？"
5. 更新待办清单
```

### 3.3 定期盘点机制

**每周自动执行（REPORT职责）：**

```
每周一报告包含：
- 本周新增文件清单
- 版本追踪表更新
- 待确认信息清单
- 遗漏风险预警
```

**每次会话开始时自动执行：**

```
开始对话时：
1. 读取待办事项.md
2. 读取版本追踪表
3. 提醒："您有X项待办、Y个待确认信息"
4. 询问："今天优先处理哪项？"
```

## 四、方法论归纳

### 4.1 信息分类方法论

| 信息类型 | 存储位置 | 管理工具 | 更新频率 |
|---------|---------|---------|---------|
| 决策记录 | /egbertie/决策记录.md | 实时记录 | 每次决策 |
| 方案版本 | /egbertie/所有方案版本/ | 版本追踪表 | 每次更新 |
| 待办事项 | /egbertie/待办事项.md | 待办清单 | 每日更新 |
| 对话记录 | /002-DAILY-LOG/ | 自动记录 | 每次对话 |
| 角色档案 | /organization/ | 角色管理 | 变更时更新 |
| 研究成果 | /research/ | 研究管理 | 每周更新 |

### 4.2 信息查找方法论

**快速查找路径：**

```
1. 不知道信息在哪 → 查总索引（000-INDEX.md）
2. 找最新版本 → 查版本追踪表（001-VERSION-TRACKING.md）
3. 找历史决策 → 查决策记录（/egbertie/决策记录.md）
4. 找待办事项 → 查待办清单（/egbertie/待办事项.md）
5. 找对话记录 → 查每日日志（/002-DAILY-LOG/）
```

### 4.3 信息更新方法论

**更新原则：**
- **一处更新，处处同步**：更新文件后，同步更新索引
- **变更必留痕**：任何变更记录到版本追踪表
- **过期必归档**：过时版本移动到archive目录
