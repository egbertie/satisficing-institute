## S1: 全局考虑 (Global Coverage) 【必须】

### 定义
覆盖人、事、物、环境、外部集成、边界情况的完整视角。

### 检查清单

| 维度 | 检查项 | 达标标准 |
|------|--------|----------|
| **人** | 干系人识别 | 列出所有受影响的人群 |
| **人** | 角色定义 | 明确各角色的职责和权限 |
| **事** | 场景覆盖 | 正常/异常/边界场景 |
| **事** | 流程完整 | 输入→处理→输出→反馈 |
| **物** | 资源需求 | 所需工具/数据/资产 |
| **物** | 交付物 | 明确的产出和验收标准 |
| **环境** | 运行环境 | 技术/组织/时间约束 |
| **环境** | 外部依赖 | 第三方服务/API依赖 |
| **边界** | 范围界定 | 明确什么不做 |
| **边界** | 异常处理 | 边界情况的处理机制 |

### 输出要求
- 必须包含干系人分析表
- 必须包含场景覆盖矩阵
- 必须包含边界情况声明

## S2: 系统考虑 (System Loop) 【必须】

### 定义
输入→处理→输出→反馈的完整闭环，而非单点功能。

### 核心流程

```
输入(Input)
    ↓
处理(Process)
    ↓
输出(Output)
    ↓
反馈(Feedback)
    ↓
回到输入（迭代优化）
```

### 检查清单

| 阶段 | 检查项 | 达标标准 |
|------|--------|----------|
| **输入** | 完整性检查 | 所有必要输入已定义 |
| **输入** | 验证机制 | 输入有效性校验 |
| **处理** | 流程清晰 | 步骤明确，无歧义 |
| **处理** | 异常处理 | 错误路径已定义 |
| **输出** | 可验证 | 明确的完成标准 |
| **输出** | 可追溯 | 输出与输入的关联 |
| **反馈** | 监控机制 | 状态和进度可追踪 |
| **反馈** | 改进机制 | 基于反馈的优化 |

### 输出要求
- 必须包含流程图或状态机
- 必须包含故障处理机制
- 必须包含反馈收集渠道

## S3: 迭代机制 (Iterative) 【必须】

### 定义
PDCA循环或类似的持续改进机制，而非一次性交付。

### PDCA循环

| 阶段 | 活动 | 输出 |
|------|------|------|
| **Plan** | 制定计划和目标 | 计划文档 |
| **Do** | 执行计划 | 执行记录 |
| **Check** | 检查结果与目标的差距 | 检查报告 |
| **Act** | 基于检查改进 | 改进方案 |

### 版本历史

| 版本 | 日期 | 变更 | 作者 |
|------|------|------|------|
| v1.0.0 | YYYY-MM-DD | 初始版本 | [作者] |
| v1.1.0 | YYYY-MM-DD | [变更描述] | [作者] |

### 反馈收集

| 来源 | 频率 | 用途 |
|------|------|------|
| 用户反馈 | 持续 | 需求调整 |
| 使用数据 | 定期 | 性能优化 |
| 错误报告 | 实时 | 缺陷修复 |

### 输出要求
- 必须包含版本历史表
- 必须包含改进机制说明
- 必须包含反馈收集渠道

## S4: Skill化 (Standardized) 【必须】

### 定义
标准化接口，可安装、可调用、可复用，而非定制化脚本。

### 标准格式

```yaml
---
name: skill-name
version: 1.0.0
description: |
  多行描述，说明Skill的功能和价值
author: [作者]
tags:
  - tag1
  - tag2
requires:
  - tool1
  - tool2
---

# Skill内容...
```

### 接口标准

| 元素 | 要求 | 示例 |
|------|------|------|
| **命令** | 统一命名 | `skill-name command` |
| **参数** | 明确类型 | `--param value` |
| **返回值** | 结构化 | JSON/YAML |
| **错误码** | 标准化 | 0=成功，非0=错误类型 |

### 输出要求
- 必须符合SKILL.md格式
- 必须包含依赖声明
- 必须包含使用示例

## S5: 自动化 (Automated) 【必须】

### 定义
cron定时任务或事件触发，无需人工干预即可运行。

### 自动化级别

| 级别 | 描述 | 示例 |
|------|------|------|
| **L1** | 定时触发 | cron每日执行 |
| **L2** | 事件触发 | 文件变更触发 |
| **L3** | 条件触发 | 满足条件时自动执行 |
| **L4** | 自愈触发 | 检测到异常自动修复 |

### Cron配置示例

```json
{
  "jobs": [
    {
      "name": "task-name",
      "schedule": "0 9 * * *",
      "command": "skill-name command",
      "description": "任务描述"
    }
  ]
}
```

### 输出要求
- 必须包含cron配置
- 必须包含监控机制
- 必须包含告警规则

## S6: 认知谦逊 (Epistemic Humility) 【推荐】

### 定义
标注信息的认知状态（KNOWN/INFERRED/UNKNOWN），承认不确定性。

### 认知状态标注

| 状态 | 含义 | 使用场景 |
|------|------|----------|
| **[KNOWN]** | 已验证的事实 | 有可靠来源的数据 |
| **[INFERRED]** | 基于数据的推断 | 合理推演但未经直接验证 |
| **[UNKNOWN]** | 信息不足 | 假设或猜测 |
| **[待验证]** | 需要确认 | 存疑内容 |

### 置信度标注

```yaml
confidence: 0.85  # 0-1之间的数值
evidence:
  - "来源1"
  - "来源2"
limitations:
  - "局限性1"
  - "局限性2"
```

### 输出要求
- 关键事实必须标注认知状态
- 推断性内容必须说明依据
- 不确定性必须明确声明

## S7: 对抗验证 (Devil's Advocate) 【推荐】

### 定义
主动提出反方观点、失效场景、质疑假设，预防确认偏误。

### 反方观点模板

```markdown
#### 观点N: [反方观点简述]
**提出者**: [角色]
**论据**:
- [论点1]
- [论点2]

**失效场景**:
```yaml
scenario: [场景名称]
trigger: [触发条件]
consequence: [后果]
```

**缓解措施**: [如何应对]
```

### 失效预警指标

```yaml
warning_indicators:
  - metric: [指标名称]
    threshold: [阈值]
    meaning: [含义]
    action: [应对措施]
```

### 输出要求
- 至少3个反方观点
- 每个观点必须有失效场景
- 必须有缓解措施
