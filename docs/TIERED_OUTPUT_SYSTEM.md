# 分级输出系统 (Tiered Output System)

> **版本**: v1.0  
> **创建时间**: 2026-03-21  
> **状态**: 生产就绪  
> **优先级**: P0

---

## 1. 系统概述

### 1.1 设计目标

分级输出系统旨在优化Token使用效率，通过三级输出长度机制，在不同场景下提供恰到好处的信息量。

**核心原则**:
- **效率优先**: 避免过度输出，节省Token消耗
- **灵活可控**: 支持用户主动选择或系统自动判断
- **渐进展开**: 保留完整版，支持需要时展开详情

### 1.2 三级输出定义

| 级别 | 名称 | Token范围 | 适用场景 | 响应时间目标 |
|------|------|-----------|----------|--------------|
| L1 | 极简版 | <50 tokens | 紧急响应、简单确认 | <1秒 |
| L2 | 标准版 | 200-500 tokens | 常规任务汇报 | <3秒 |
| L3 | 详细版 | 1000+ tokens | 深度分析、复杂任务 | <10秒 |

## 2. 分级标准详细规范

### 2.1 L1 - 极简版 (一句话)

**定义**: 用最精炼的语言传达核心结论和单一行动项

**Token限制**: <50 tokens

**适用场景**:
- P0紧急任务确认
- 状态快速查询
- 简单是/否问题
- Token预算<30%时的自动降级

**输出结构**:
```
[结论] + [一个行动项]
```

**模板示例**:
```
✅ 任务已完成。建议：检查邮件确认最终交付。
⚠️ 遇到阻塞。需要：您提供API密钥才能继续。
✅ 部署成功。下一步：监控日志30分钟确认稳定。
```

---

### 2.2 L2 - 标准版 (一段话)

**定义**: 提供摘要、关键发现和下一步行动的平衡信息

**Token限制**: 200-500 tokens

**适用场景**:
- P1/P2常规任务汇报
- 进度同步
- 问题诊断与建议
- 用户未指定输出长度时的默认级别

**输出结构**:
```
## 摘要
[一句话总结核心内容]

## 关键发现
- [发现1]: [简要说明]
- [发现2]: [简要说明]
- [发现3]: [简要说明]

## 下一步
1. [行动项1] - [负责人/时间]
2. [行动项2] - [负责人/时间]
```

**模板示例**:
```markdown
## 摘要
完成3个文件审计，发现2个高风险项需立即处理。

## 关键发现
- **文件完整性**: 95%通过，缺失1个配置文件
- **安全风险**: 发现2个硬编码密钥，需轮换
- **性能优化**: 1个查询可优化，预计提升30%效率

## 下一步
1. 立即轮换密钥 - 安全团队/2小时内
2. 补充配置文件 - 开发团队/今日完成
3. 部署查询优化 - 明日上线
```

**Token限制**: 1000+ tokens

**适用场景**:
- P0复杂任务深度分析
- 战略决策支持
- 根因分析报告
- 用户明确要求详细说明 (/detail)

**输出结构**:
```
# [报告标题]

## 执行摘要
[3-5句话概括全文核心结论]

## 背景与上下文

## 可选方案
### 方案A: [名称]
- **优点**: [列表]
- **缺点**: [列表]
- **成本**: [估算]
- **风险**: [等级]
```

## 3. 自动分级触发器

### 3.1 用户明确指令

| 指令 | 触发级别 | 说明 |
|------|----------|------|
| `/brief` 或 `/b` | L1 | 强制极简输出 |
| `/normal` 或 `/n` | L2 | 标准输出（默认） |
| `/detail` 或 `/d` | L3 | 强制详细输出 |

**使用示例**:
```
用户: /brief 检查今日任务状态
→ 触发L1输出

用户: /detail 分析系统性能瓶颈
→ 触发L3输出
```

### 3.2 任务紧急度自动判断

| 优先级 | 默认级别 | 说明 |
|--------|----------|------|
| P0 | L1/L2 | 紧急任务，优先速度，但复杂分析用L3 |
| P1 | L2 | 高优先级，标准输出 |
| P2 | L2 | 中等优先级，标准输出 |
| P3 | L1 | 低优先级，极简输出节省Token |

**自动调整规则**:
- P0 + 复杂任务描述 → L2（平衡速度与信息）
- P0 + "分析"/"报告"关键词 → L3（必须详细）
- P3 + 简单查询 → L1（最大节省）

### 3.3 Token预算状态触发

| 预算状态 | 触发行为 | 说明 |
|----------|----------|------|
| >70% | 按正常规则 | 无限制 |
| 30%-70% | 默认降级至L2 | 轻度节省 |
| <30% | 强制L1 | 紧急节省模式 |
| <10% | L1 + 警告 | 临界模式 |

**Token预算检查逻辑**:
```yaml
check_token_budget:
  - if: budget_remaining > 70%
    action: follow_normal_rules
  - if: 30% <= budget_remaining <= 70%
    action: default_to_L2
    message: "Token预算适中，已切换至标准输出模式"
  - if: budget_remaining < 30%
    action: force_L1
    message: "⚠️ Token预算<30%，已启用极简输出模式"
  - if: budget_remaining < 10%
    action: force_L1 + alert
    message: "🚨 Token预算临界！仅提供核心结论"
```

### 3.4 用户历史偏好学习

**学习维度**:
1. **默认偏好**: 用户最常使用的输出级别
2. **场景映射**: 特定类型任务偏好的级别
3. **时间模式**: 不同时段可能的偏好（如深夜偏好L1）

**学习规则**:
```yaml
learn_user_preference:
  tracking:
    - user_id
    - task_type
    - requested_level
    - actual_level_used
    - user_feedback_score
  
  adaptation:
    - if: 用户连续3次手动选择L1
      then: 该任务类型默认L1
    - if: 用户反馈"太详细"超过2次
      then: 该场景降级一档
    - if: 用户追问"还有呢"超过2次
      then: 该场景升级一档
```

## 4. 系统集成方案

### 4.1 响应生成逻辑修改

**原始流程**:
```
用户输入 → 处理 → 生成完整回复 → 输出
```

**新流程**:
```
用户输入 → 解析意图 → 确定输出级别 → 按模板生成 → 输出
                  ↓
            触发器评估:
            - 用户指令?
            - 优先级标签?
            - Token预算?
            - 历史偏好?
```

### 4.2 输出长度控制实现

**配置层** (`skills/tiered-output/config.yaml`):
```yaml
tier_limits:
  L1:
    max_tokens: 50
    max_sentences: 2
    max_lines: 1
  L2:
    max_tokens: 500
    max_sections: 3
    max_bullet_points: 6
  L3:
    min_tokens: 1000
    require_sections: ["摘要", "分析", "建议"]
```

**执行层**:
```python
def generate_response(user_input, context):
    # 1. 确定输出级别
    tier = determine_output_tier(user_input, context)
    
    # 2. 获取对应模板
    template = get_template(tier)
    
    # 3. 生成内容（限制在Token预算内）
    content = generate_with_limit(tier, template, context)
    
    # 4. 添加展开选项（L1/L2时）
    if tier in ["L1", "L2"]:
        content += generate_expand_option(tier)
    
    return content
```

### 4.3 完整版保留与展开机制

**L1/L2输出时的附加选项**:

```markdown
---
💡 **需要更详细的信息？**
- 回复 "/expand" 查看完整分析
- 回复 "/detail" 获取L3详细版
```

**展开实现**:
```python
def handle_expand_request(user_input, conversation_history):
    if user_input == "/expand":
        # 找到上一条L1/L2回复对应的L3版本
        full_version = retrieve_full_version(conversation_history[-1])
        return full_version
```
