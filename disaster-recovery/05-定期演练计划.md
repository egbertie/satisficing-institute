# 灾备定期演练计划 V1.0

> **文档版本**: 1.0  
> **生效日期**: 2026-03-12  
> **演练频率**: 季度完整演练 + 月度检查  
> **参考对象**: Netflix Chaos Engineering + Google Disaster Recovery Testing

---

## 1. 演练概述

### 1.1 目的

灾难恢复演练不是为了 "通过测试"，而是为了 **发现预案中的漏洞**。

> "如果你不定期测试恢复流程，你就没有灾备方案 —— 你只是有希望。"
> —— 某SRE格言

### 1.2 演练类型

| 类型 | 频率 | 时长 | 范围 | 目的 |
|-----|------|------|------|------|
| **桌面推演** | 每月 | 30分钟 | 文档审查 | 保持熟悉度 |
| **功能验证** | 每月 | 1小时 | 备份验证 | 确保备份可用 |
| **模拟演练** | 每季度 | 4小时 | 单场景恢复 | 验证流程可行性 |
| **全面演练** | 每半年 | 1天 | 完整恢复 | 验证RTO/RPO目标 |
| **混沌测试** | 随机 | 不定 | 随机故障 | 发现意外依赖 |

## 4. 季度演练剧本

### 4.1 Q1演练: GitHub仓库完全丢失恢复

**日期**: 2026-03-15 (周六)  
**时长**: 4小时  
**场景**: 模拟GitHub账号被封，从镜像源恢复所有代码

#### 演练前准备

- [ ] 选择测试仓库 (不影响生产)
- [ ] 备份当前GitHub设置 (webhooks, secrets)
- [ ] 通知相关方 (演练期间不要惊慌)
- [ ] 准备计时器

#### 演练剧本

**Act 1: 场景注入 (30分钟)**
```
[09:00] 场景开始
• 模拟: GitHub账号收到封禁通知
• 操作: 临时修改hosts屏蔽github.com
• 验证: 确认无法访问GitHub

检查点:
□ 能否确认是"GitHub问题"而非网络问题?
□ 能否快速找到灾备手册?
```

**Act 2: 应急响应 (30分钟)**
```
[09:30] 启动恢复流程
• 操作: 按照操作手册3.2章节执行
• 决策: 使用Gitee镜像还是本地备份?

检查点:
□ RTO时钟开始计时
□ 相关通知已发送
```

**Act 3: 数据恢复 (2小时)**
```
[10:00] 执行恢复操作
• 从Gitee克隆镜像仓库
• 推送到新的GitHub账号/组织
• 恢复 Issues/PRs (从备份)

详细步骤:
1. git clone --mirror https://gitee.com/backup/repo.git
2. 创建新GitHub仓库
3. git push --mirror https://github.com/new/repo.git
4. 验证提交历史完整

检查点:
□ 所有分支已恢复
□ 所有标签已恢复
□ 提交历史完整
```

**Act 4: 验证与复盘 (1小时)**
```
[12:00] 恢复验证
• 克隆恢复的仓库到新目录
• 运行测试套件验证代码
• 检查关键文件存在

复盘问题:
1. 实际RTO是多少? 与目标4小时对比如何?
2. 哪些步骤比预期困难?
3. 发现了什么依赖关系 (如CI/CD webhook)?
4. 文档中哪些地方需要更新?
```

### 4.3 H1全面演练: 完整环境重建

**日期**: 2026-06-14 (周六)  
**时长**: 全天 (8小时)  
**场景**: 模拟笔记本被盗，从裸机重建完整开发环境

#### 演练前准备

- [ ] 准备测试用裸机/虚拟机
- [ ] 确保所有备份最新
- [ ] 准备离线备份介质
- [ ] 安排好不受打扰的时间

#### 演练流程

| 时间 | 阶段 | 任务 | 成功标准 |
|-----|------|------|---------|
| 09:00-10:00 | 准备 | 模拟"新机"环境 | 干净的系统 |
| 10:00-12:00 | 系统 | 基础环境恢复 | 开发工具安装完成 |
| 13:00-15:00 | 数据 | 代码仓库恢复 | 所有项目可编译 |
| 15:00-16:00 | 配置 | 环境变量/密钥恢复 | 能连接生产环境 |
| 16:00-17:00 | 验证 | 完整功能测试 | 能正常开发工作 |

#### 关键验证点

```bash
# 开发环境验证脚本
#!/bin/bash

echo "=== 灾备恢复验证 ==="

# 1. Git可用且配置了身份
git --version && git config user.name && git config user.email

# 2. 代码仓库可克隆
for repo in repo1 repo2 repo3; do
    git clone https://github.com/org/$repo /tmp/test_$repo
done

# 3. 能连接服务器
ssh -o ConnectTimeout=5 server "echo 'SSH OK'"

# 4. 开发工具可运行
node --version
go version
docker --version

# 5. 能构建项目
cd /tmp/test_repo1 && npm install && npm build

echo "=== 验证完成 ==="
```

## 5. 混沌测试 (随机演练)

### 5.1 混沌测试理念

> "在可控范围内制造故障，验证系统的韧性。"

### 5.2 随机故障卡片

```
═══════════════════════════════════════════════════════════════
               🎲 混沌测试故障卡片
═══════════════════════════════════════════════════════════════

使用方式:
1. 随机抽取一张卡片
2. 不提前准备，立即执行
3. 记录发现和解决时间

卡片列表:
─────────

♠ A  突然断开GitHub访问30分钟
     任务: 继续开发工作并记录效率损失

♠ 2  删除一个本地仓库，从备份恢复
     任务: 测量恢复所需时间

♠ 3  禁用一个API密钥8小时
     任务: 切换到备用方案继续工作

♠ 4  模拟Notion离线24小时
     任务: 使用本地备份继续项目

♠ 5  "丢失"SSH密钥
     任务: 从加密备份恢复并重新部署

♠ 6  删除最新的Git提交历史
     任务: 使用reflog或备份恢复

♠ 7  模拟密码管理器不可用
     任务: 使用离线备份访问关键服务

♠ 8  故意"损坏"一个重要文件
     任务: 从版本历史恢复

♠ 9  随机选择一个备份，验证可恢复性
     任务: 完整解压并检查内容

♠ 10 模拟云服务中断，切换到本地方案
     任务: 验证离线工作流

═══════════════════════════════════════════════════════════════
```

### 5.3 执行建议

- **频率**: 每季度随机执行1-2次
- **时机**: 选择工作相对轻松的时期
- **范围**: 只影响测试环境或个人项目
- **记录**: 每次混沌测试后记录发现

### 7.2 演练脚本自动化

```bash
#!/bin/bash
# 演练辅助脚本: drill-helper.sh

DRILL_TYPE="${1:-quarterly}"
DATE=$(date +%Y%m%d)
LOG_FILE="logs/drill-${DATE}.log"

mkdir -p logs

echo "=== 灾备演练辅助脚本 ===" | tee -a "$LOG_FILE"
echo "类型: $DRILL_TYPE" | tee -a "$LOG_FILE"
echo "开始时间: $(date)" | tee -a "$LOG_FILE"
echo "" | tee -a "$LOG_FILE"

# 演练前快照
echo "【演练前状态快照】" | tee -a "$LOG_FILE"
echo "GitHub镜像状态:" | tee -a "$LOG_FILE"
ls -la ~/Backups/GitHub/ | tail -5 | tee -a "$LOG_FILE"

echo "Notion备份状态:" | tee -a "$LOG_FILE"
ls -la ~/Backups/Notion/*.tar.gz 2>/dev/null | tail -3 | tee -a "$LOG_FILE"

echo "" | tee -a "$LOG_FILE"
echo "计时开始..." | tee -a "$LOG_FILE"
echo "按回车键标记阶段完成，输入 'done' 结束演练"

read -p "阶段1完成? (回车继续): "
echo "阶段1完成: $(date)" | tee -a "$LOG_FILE"

read -p "阶段2完成? (回车继续): "
echo "阶段2完成: $(date)" | tee -a "$LOG_FILE"

read -p "阶段3完成? (回车继续): "
echo "阶段3完成: $(date)" | tee -a "$LOG_FILE"

echo "" | tee -a "$LOG_FILE"
echo "演练结束: $(date)" | tee -a "$LOG_FILE"
echo "日志保存至: $LOG_FILE"
```
