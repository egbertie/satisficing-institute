## 2. 通用恢复流程

### 2.1 标准响应流程 (STOP)

```
┌───────────────────────────────────────────────────────────┐
│  S - Stop    立即停止操作，不要慌乱中误操作加重损失        │
│  T - Think   评估影响范围，确定灾难等级                    │
│  O - Observe 收集信息，查看日志和监控                      │
│  P - Proceed 按照手册执行恢复操作                          │
└───────────────────────────────────────────────────────────┘
```

### 2.2 恢复流程模板

**阶段1: 评估 (0-15分钟)**
- [ ] 确认影响范围 (哪些服务受影响)
- [ ] 确认影响程度 (完全不可用/部分受损)
- [ ] 确定灾难等级 (P0/P1/P2/P3)
- [ ] 通知相关人员

**阶段2: 稳定 (15-60分钟)**
- [ ] 启动备用方案 (如果有)
- [ ] 隔离问题防止扩散
- [ ] 开始数据恢复准备

**阶段3: 恢复 (按RTO执行)**
- [ ] 执行具体恢复操作
- [ ] 验证恢复结果
- [ ] 切换回主服务

**阶段4: 复盘 (恢复后24小时内)**
- [ ] 记录事件时间线
- [ ] 分析根本原因
- [ ] 更新预案防止复发

## 3. 场景化恢复指南

### 3.1 GitHub 服务中断

**症状**: github.com 无法访问 / 推送失败

**影响**: 代码无法推送/拉取，CI/CD 可能中断

**RTO**: 核心服务 4 小时内恢复开发工作流

#### 检查清单

- [ ] 确认 GitHub 状态: https://www.githubstatus.com
- [ ] 检查本地网络连接
- [ ] 确认是否为区域性故障

#### 恢复操作

**选项A: 切换到镜像源 (推荐)**

```bash
# 1. 查看当前remote
cd your-project
git remote -v

# 2. 切换到Gitee镜像 (如果已设置)
git remote set-url origin https://gitee.com/yourname/project.git

# 3. 验证推送/拉取
git pull
git push
```

**选项B: 纯本地工作**

```bash
# 继续使用本地仓库
git commit -am "wip: GitHub中断期间的更改"

# 等GitHub恢复后推送
git push origin main
```

**选项C: 使用GitHub Enterprise Server (如有)**
```bash
# 切换到GHE实例
git remote set-url origin https://ghe.company.com/yourname/project.git
```

#### 事后操作

GitHub恢复后：
- [ ] 将remote改回 GitHub
- [ ] 同步镜像源与主仓库
- [ ] 更新服务状态监控

---

### 3.2 GitHub 仓库数据异常

**症状**: 仓库为空 / 提交历史丢失 / 分支损坏

**影响**: 代码历史丢失，协作受阻

**RTO**: 2小时内恢复仓库完整状态

#### 检查清单

- [ ] 确认是单个仓库还是多个
- [ ] 检查 GitHub Issues 是否有其他人报告
- [ ] 确认本地是否有完整副本

#### 恢复操作

**场景A: 本地有完整仓库**

```bash
# 1. 进入本地仓库
cd your-project

# 2. 确保有所有分支的最新代码
for branch in $(git branch -r | grep -v HEAD); do
    git checkout --track $branch 2>/dev/null || true
done

# 3. 强制推送恢复GitHub
for branch in $(git branch | sed 's/^\*//'); do
    git push origin $branch --force
done

# 4. 推送标签
git push origin --tags --force
```

**场景B: 从镜像源恢复**

```bash
# 1. 克隆镜像
mkdir -p /tmp/recovery
cd /tmp/recovery
git clone --mirror https://gitee.com/yourname/project.git

# 2. 推送到GitHub
cd project.git
git push --mirror https://github.com/yourname/project.git

# 3. 验证
open https://github.com/yourname/project
```

**场景C: 从本地备份恢复**

```bash
# 1. 解压备份
cd ~/Backups/GitHub
tar xzf project_20260312_020000.tar.gz

# 2. 恢复
cd project
git push origin --all --force
git push origin --tags --force
```

#### 事后操作

- [ ] 验证所有提交历史完整
- [ ] 检查分支保护设置
- [ ] 重新配置CI/CD Webhooks
- [ ] 通知团队成员检查本地仓库

### 3.3 Notion 服务中断

**症状**: notion.so 无法访问 / 同步失败

**影响**: 文档协作中断

**RTO**: 8小时内恢复文档访问

#### 检查清单

- [ ] 确认 Notion 状态: https://status.notion.so
- [ ] 检查离线缓存是否可用 (Notion Desktop)

#### 恢复操作

**选项A: 使用离线缓存**

Notion Desktop 会自动缓存最近访问的页面：
1. 不要退出 Notion 应用
2. 继续使用离线模式编辑
3. 等恢复后自动同步

**选项B: 切换到本地备份**

```bash
# 1. 找到最近备份
cd ~/Backups/Notion
ls -la notion_backup_*.tar.gz | tail -5

# 2. 解压查看
tar tzf notion_backup_20260312.tar.gz | head -20

# 3. 解压使用
tar xzf notion_backup_20260312.tar.gz
# 使用 Markdown 文件继续工作
```

**选项C: 使用备用协作工具**
- 飞书文档 (如果已同步)
- 本地 Markdown 文件
- Google Docs (临时)

### 3.4 Notion 数据误删除

**症状**: 页面被误删 / 数据库被清空

**影响**: 重要信息丢失

**RTO**: 根据重要性 30分钟-4小时

#### 检查清单

- [ ] 确认删除时间 (影响恢复方式)
- [ ] 检查 Notion Trash (30天内可恢复)
- [ ] 检查本地备份时间

#### 恢复操作

**选项A: 从 Notion Trash 恢复 (删除<30天)**

1. 打开 Notion → 左侧边栏 → Trash
2. 找到删除的页面
3. 点击 Restore

**选项B: 从历史版本恢复**

1. 打开页面
2. 点击右上角 ••• → Page history
3. 选择删除前的版本
4. 点击 Restore this version

**选项C: 从备份导入**

```bash
# 1. 解压备份
cd ~/Backups/Notion
tar xzf notion_backup_$(date +%Y%m%d).tar.gz

# 2. 手动导入Notion
# 创建新页面 → Import → 上传Markdown/HTML
```

### 3.5 本地开发环境崩溃

**症状**: 电脑无法开机 / 系统损坏 / 硬盘故障

**影响**: 所有本地工作暂停

**RTO**: 24小时内恢复开发环境

#### 检查清单

- [ ] 确认硬件还是软件问题
- [ ] 确认数据是否可从硬盘读取
- [ ] 准备备用设备

#### 恢复操作

**阶段1: 数据抢救 (如果硬盘可读)**

```bash
# 使用Live USB启动
# 挂载原硬盘
sudo mount /dev/sda1 /mnt

# 复制关键数据到外置硬盘
cp -r /mnt/home/username/workspace /media/backup/
cp -r /mnt/home/username/.ssh /media/backup/
```

**阶段2: 环境重建**

```bash
# 1. 系统安装
# - 重装 macOS / Ubuntu / Windows

# 2. 运行恢复脚本
cd ~/Backups/Local
tar xzf local_backup_20260312.tar.gz

# 3. 执行恢复
cd local_backup_20260312*/
./restore.sh

# 4. 重新克隆仓库
cat git-repos.txt | while read repo; do
    if [[ -n "$repo" && ! "$repo" =~ ^# ]]; then
        git clone $repo
done
```

**阶段3: 开发工具恢复**

```bash
# macOS Homebrew
brew bundle --file=packages/Brewfile

# VSCode 扩展
cat packages/vscode-extensions.txt | xargs -n 1 code --install-extension

# 恢复SSH密钥 (从加密备份)
gpg -d keys/ssh_keys_20260312.tar.gz.gpg > ssh_keys.tar.gz
tar xzf ssh_keys.tar.gz -C ~/.ssh/
chmod 600 ~/.ssh/id_*
```

### 3.6 勒索软件攻击

**症状**: 文件被加密 / 出现勒索信 / 异常后缀名

**影响**: 所有数据被加密

**RTO**: 48小时内隔离并恢复

#### ⚠️ 关键警告

**DO NOT:**
- ❌ 支付赎金 (不保证解密，且鼓励犯罪)
- ❌ 运行不明程序
- ❌ 连接网络共享

**DO:**
- ✅ 立即断网断电
- ✅ 保留被加密文件 (可能有解密工具)
- ✅ 报警并联系专业安全团队

#### 恢复操作

**阶段1: 隔离 (立即)**

```bash
# 1. 立即断网
# - 拔掉网线
# - 关闭WiFi

# 2. 关机 (不要正常关机，直接断电)
# 长按电源键强制关机
```

**阶段2: 评估损失**

1. 使用干净Live USB启动
2. 检查哪些文件被加密 (常见后缀: .locked, .encrypted, .crypt)
3. 拍照记录勒索信息

**阶段3: 从离线备份恢复**

```bash
# 1. 使用干净的备用电脑或重装系统

# 2. 从离线备份恢复 (加密U盘/外置硬盘)
# 不要连接可能被感染的备份

# 3. 验证备份干净后再恢复
```

**阶段4: 安全检查**

- [ ] 全盘杀毒
- [ ] 检查计划任务和启动项
- [ ] 更改所有密码
- [ ] 启用2FA
- [ ] 检查日志找入侵点

### 3.7 账号被盗/被封

**症状**: 无法登录 / 收到异常邮件 / 发现未授权操作

**影响**: 数据控制权丧失

**RTO**: 4小时内恢复账号访问

#### 检查清单

- [ ] 确认是密码错误还是账号被封
- [ ] 检查邮箱是否有安全警报
- [ ] 尝试密码重置

#### 恢复操作

**GitHub 账号恢复**

```bash
# 1. 尝试密码重置
open https://github.com/password_reset

# 2. 如被封禁，联系支持
open https://github.com/contact

# 3. 启用2FA (恢复后)
# Settings → Security → Two-factor authentication

# 4. 检查授权应用
# Settings → Applications → Authorized OAuth Apps
# 撤销可疑授权
```

**Notion 账号恢复**

```bash
# 1. 密码重置
open https://www.notion.so/login#reset-password

# 2. 检查工作空间成员
# Settings & Members → Members
# 移除可疑成员

# 3. 导出数据备份 (恢复后立即执行)
# Settings & Members → Export → Export all workspace content
```

**预防措施 (恢复后执行)**

- [ ] 密码管理器生成强密码
- [ ] 所有关键服务启用2FA
- [ ] 检查并撤销可疑OAuth授权
- [ ] 审查最近登录历史
