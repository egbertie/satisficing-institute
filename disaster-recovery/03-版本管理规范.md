# 灾备版本管理规范 V1.0

> **文档版本**: 1.0  
> **生效日期**: 2026-03-12  
> **适用范围**: 所有灾备相关文档、脚本、配置  
> **参考对象**: Semantic Versioning 2.0.0 + Git Flow 工作流

---

## 1. 概述

### 1.1 目的
本文档定义灾备方案中所有交付物的版本管理规范，确保：
- 版本号语义清晰，一眼识别变更性质
- 变更历史可追溯，便于问题排查
- 多版本并行管理，支持灰度恢复
- 版本与备份关联，快速定位正确版本

### 1.2 适用范围
| 类别 | 示例 | 版本管理 |
|-----|------|---------|
| 策略文档 | 灾备策略文档 | ✅ 语义化版本 |
| 操作手册 | 恢复操作手册 | ✅ 语义化版本 |
| 自动化脚本 | 备份脚本套件 | ✅ 语义化版本 |
| 配置文件 | 环境变量模板 | ✅ 语义化版本 |
| 演练记录 | 演练报告 | ✅ 日期版本 |
| 备份数据 | 仓库/文档备份 | ✅ 日期+版本标签 |

## 2. 版本号规范 (Semantic Versioning)

### 2.1 基本格式

```
主版本号.次版本号.修订号[-预发布标识]
  │        │       │        │
  │        │       │        └── 可选: alpha, beta, rc
  │        │       └────────── 修订版本: 向下兼容的问题修正
  │        └────────────────── 次版本: 向下兼容的功能新增
  └─────────────────────────── 主版本: 不兼容的重大变更

示例: 1.2.3, 2.0.0-beta, 1.5.0-alpha.1
```

### 2.2 版本号递增规则

| 变更类型 | 示例 | 版本变化 |
|---------|------|---------|
| 文档错别字修正 | 修复typo | 1.2.3 → 1.2.4 |
| 新增操作步骤 | 添加恢复验证环节 | 1.2.3 → 1.3.0 |
| 重构架构 | 从单机备份改为分布式 | 1.2.3 → 2.0.0 |
| 新增备份目标 | 支持飞书文档备份 | 1.2.3 → 1.3.0 |
| 修改RTO/RPO | 核心指标调整 | 1.2.3 → 2.0.0 |
| 预发布测试 | 新方案验证 | 1.3.0 → 2.0.0-alpha.1 |

### 2.3 版本发布生命周期

```
                    发现严重bug
                       │
    开发中 ──► 预发布 ──┴──► 正式发布 ──► 维护期 ──► 退役
      │         │            │           │          │
      ▼         ▼            ▼           ▼          ▼
   alpha      beta          rc        patch      archive
   (内测)    (公测)       (候选)     (补丁)      (归档)
```

## 3. 灾备专用版本命名

### 3.1 文档版本

| 文档类型 | 版本示例 | 说明 |
|---------|---------|------|
| 灾备策略 | `DR-Strategy-v2.1.0` | 主策略文档 |
| 操作手册 | `DR-Runbook-v1.5.3` | 恢复操作手册 |
| 演练计划 | `DR-Drill-2026-Q1` | 季度演练计划 |
| 演练报告 | `DR-Drill-Report-20260312` | 具体日期 |

### 3.2 脚本版本

```bash
#!/bin/bash
################################################################################
# 灾备自动化备份脚本套件 - 主控脚本
# 
# 版本: 1.2.1
# 日期: 2026-03-12
# 变更: 修复阿里云OSS同步失败问题

DR_VERSION="1.2.1"
DR_VERSION_DATE="2026-03-12"
DR_VERSION_CHANGELOG="修复阿里云OSS同步失败问题"
```

## 4. 变更管理流程

### 4.1 变更请求 (CR) 流程

```
┌─────────────────────────────────────────────────────────┐
│ 1. 提交变更请求                                          │
│    - 描述变更内容                                        │
│    - 说明影响范围                                        │
│    - 评估风险等级                                        │
└─────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────┐
│ 2. 影响评估                                              │
│    - RTO/RPO 是否变化?                                   │
│    - 是否需要重新演练?                                   │
│    - 版本号如何递增?                                     │
└─────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────┐
│ 3. 审批决策                                              │
│    - 文档变更: 技术负责人审批                             │
│    - 脚本变更: 测试通过后方可合并                         │
│    - 策略变更: 需演练验证                                 │
└─────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────┐
│ 4. 实施变更                                              │
│    - 更新文档/代码                                       │
│    - 更新版本号                                          │
│    - 记录变更日志                                        │
└─────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────┐
│ 5. 验证发布                                              │
│    - 文档: 技术评审                                       │
│    - 脚本: 自动化测试                                     │
│    - 更新版本索引                                         │
└─────────────────────────────────────────────────────────┘
```

### 4.2 变更日志格式

每个文档/脚本必须包含变更日志部分：

```markdown
## 变更日志 (Changelog)

### [2.1.0] - 2026-03-12
#### 新增
- 添加阿里云OSS作为备份目标
- 增加备份数据加密选项

#### 修复
- 修复GitHub API限流处理问题

### [2.0.1] - 2026-02-28
#### 修复
- 修正RTO计算中的时区问题

### [2.0.0] - 2026-02-15
#### 重大变更
- 重构备份架构，支持多地域存储
- RPO目标从24小时缩短至4小时
```

### 4.3 Git 工作流

```
main (生产分支)
  │
  ├──► release/v2.1.0 (发布分支)
  │      │
  │      ├──► 测试验证
  │      │
  │      └──► 合并到main，打标签 v2.1.0
  │
  ├──► hotfix/v2.0.1 (热修复分支)
  │      │
  │      └──► 紧急修复后合并
  │
  └──► develop (开发分支)
         │
         ├──► feature/notion-export (功能分支)
         │
         └──► feature/oss-backup (功能分支)
```

**分支命名规范:**
- 功能分支: `feature/[描述]-v[版本]` 示例: `feature/notion-backup-v1.3`
- 修复分支: `fix/[问题描述]` 示例: `fix/github-rate-limit`
- 发布分支: `release/v[版本]` 示例: `release/v2.1.0`
- 热修复: `hotfix/v[版本]` 示例: `hotfix/v2.0.1`

## 5. 版本兼容性

### 5.1 向后兼容承诺

| 版本变化 | 兼容性保证 | 恢复说明 |
|---------|----------|---------|
| 修订版本 (1.2.3 → 1.2.4) | 100%兼容 | 可直接使用 |
| 次版本 (1.2.3 → 1.3.0) | 功能兼容 | 可能需要更新环境变量 |
| 主版本 (1.2.3 → 2.0.0) | 可能不兼容 | 必须阅读迁移指南 |

### 5.2 版本迁移指南

当主版本升级时，必须提供迁移指南：

```markdown
## 从 v1.x 迁移到 v2.0

### 破坏性变更
1. 配置文件格式从INI改为YAML
2. 环境变量前缀从 `DR_` 改为 `DR_BACKUP_`
3. 移除对Python 2的支持

### 迁移步骤
1. 备份当前配置: `cp config.ini config.ini.bak`
2. 运行迁移脚本: `./migrate-v1-to-v2.sh`
3. 验证配置: `./backup-master.sh --dry-run`
4. 执行测试备份: `./backup-master.sh --github`

### 回滚方案
如果迁移失败，恢复备份配置即可回滚到v1.x
```

```markdown

### 文档审核
- [ ] 版本号已正确更新
- [ ] 变更日志已填写
- [ ] 日期已更新
- [ ] 过期警告已移除

### 代码审核
- [ ] 版本变量已更新
- [ ] 单元测试通过
- [ ] 集成测试通过
- [ ] 无调试代码残留

### 备份验证
- [ ] 新版本脚本可正常执行
- [ ] 生成的备份文件有效
- [ ] 恢复流程可正常使用
- [ ] 版本索引已更新

### 发布步骤
- [ ] Git标签已创建: `git tag -a v2.1.0 -m "Release v2.1.0"`
- [ ] 版本索引已更新
- [ ] 团队已通知
```
