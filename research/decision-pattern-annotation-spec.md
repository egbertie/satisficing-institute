# 决策模式标注规范

> 借鉴AlphaGo特征设计，建立结构化决策标注体系
> 目标：从22年经验中提取可学习的决策模式

---

## 一、标注目标

### 1.1 短期目标（本周）

- 完成20个案例的决策点标注
- 预计提取50-80个决策点
- 建立基础标注规范

### 1.2 中期目标（1个月）

- 建立100个决策点的训练集
- 验证标注规范的完整性
- 训练初步的模式识别模型

### 1.3 长期目标（3个月）

- 建立500个决策点的高质量数据集
- 支持决策引擎的训练
- 形成可复制的标注流程

## 二、标注维度（10个核心维度）

### 2.1 维度总览

| 维度 | 说明 | 示例 | 数据类型 |
|------|------|------|----------|
| 决策情境 | 场景描述 | "银行时期，客户邀请创业" | 文本 |
| 信息完备度 | 已知/未知信息比例 | 0.3 | 0-1浮点数 |
| 时间压力 | 决策时间限制 | "48小时内决定" | 枚举 |
| 利益冲突 | 各方利益点 | "客户想拉我入伙，我担心合规" | 文本 |
| 认知偏差 | 前景理论偏差类型 | 损失厌恶、禀赋效应 | 标签列表 |
| 直觉信号 | 身体/情绪反应 | "胃不舒服，直觉觉得不对" | 文本 |
| 理性分析 | 逻辑推理过程 | "SWOT分析，风险矩阵" | 文本 |
| 最终决策 | 选择及理由 | "拒绝，时机不成熟" | 文本 |
| 结果反馈 | 后续发展 | "半年后项目失败" | 文本 |
| 模式提炼 | 通用规律 | "直觉不适+时间压力=高风险" | 文本 |

### 2.2 维度详解

#### 2.2.1 决策情境 (Context)

```yaml
描述: 决策发生的完整场景
要素:
  - 时间背景: 决策发生时的时间节点
  - 空间背景: 物理/组织环境
  - 人物关系: 涉及的关键人物
  - 触发事件: 导致决策需求的事件
  - 决策类型: 是/否/选择/退出

示例:
  "2018年3月，银行核心客户（某科技公司CEO）
   邀请我参与其AI创业项目，担任联合创始人。
   当时我还在银行任职，有48小时决定。"
```

#### 2.2.2 信息完备度 (Information_Completeness)

```yaml
描述: 已知信息占总信息的比例
计算: 已知维度数 / 决策所需总维度数
取值: 0.0 - 1.0

评估标准:
  0.0-0.3: 极低（主要靠直觉）
  0.3-0.5: 低（信息不完整）
  0.5-0.7: 中等（基本信息具备）
  0.7-0.9: 高（信息较充分）
  0.9-1.0: 极高（接近完全信息）

示例:
  已知: 客户信任度、行业前景、大致团队规模
  未知: 详细财务状况、核心技术细节、其他合伙人背景
  评估: 0.3
```

#### 2.2.3 时间压力 (Time_Pressure)

```yaml
描述: 决策时间限制的紧迫程度
取值:
  - 高: 24小时内必须决定
  - 中: 1-7天内决定
  - 低: 7天以上可决定

影响因素:
  - 机会窗口: 机会是否转瞬即逝
  - 竞争态势: 是否有竞争者
  - 对方预期: 对方的耐心程度

示例: "高 - 客户要求48小时内答复"
```

#### 2.2.4 利益冲突 (Interest_Conflict)

```yaml
描述: 各方利益点的冲突情况
分析维度:
  - 我方利益: 决策者关注的价值点
  - 对方利益: 其他方的核心诉求
  - 潜在冲突: 利益不一致的地方
  - 协调可能性: 是否存在双赢方案

示例:
  我方利益: 合规性、职业安全、长期发展
  对方利益: 快速启动、资源整合、我的银行背景
  冲突点: 合规要求 vs 快速启动需求
```

#### 2.2.5 认知偏差 (Cognitive_Bias)

```yaml
描述: 前景理论中的各类认知偏差
类型（多选）:
  - 损失厌恶: 对损失的敏感度高于收益
  - 禀赋效应: 高估已拥有事物的价值
  - 现状偏见: 倾向于维持现状
  - 锚定效应: 过度依赖第一个信息
  - 确认偏误: 寻找支持已有观点的证据
  - 可得性启发: 依赖容易想到的例子
  - 代表性启发: 以相似性判断概率
  - 控制幻觉: 高估对结果的控制力

严重程度: 高/中/低
示例:
  - 损失厌恶: 高（怕失去银行稳定工作）
  - 禀赋效应: 中（高估客户关系价值）
  - 现状偏见: 高（倾向维持现状）
```

#### 2.2.6 直觉信号 (Intuition_Signal)

```yaml
描述: 身体或情绪层面的直觉反应
记录要素:
  - 身体反应: 胃部不适、心跳加速、肌肉紧张等
  - 情绪反应: 焦虑、兴奋、不安、平静等
  - 直觉描述: "感觉不对"、"就是这个人"等
  - 强度: 强/中/弱

示例:
  身体反应: "胃部不适，隐隐作痛"
  情绪反应: "焦虑，有种说不出的不安"
  直觉描述: "总觉得哪里不对，但又说不上来"
  强度: 强
```

#### 2.2.7 理性分析 (Rational_Analysis)

```yaml
描述: 逻辑推理和结构化分析过程
方法（多选）:
  - SWOT分析
  - 风险矩阵
  - 决策树
  - 成本收益分析
  - 专家咨询
  - 对标分析
  - 情景规划

分析结论: 理性分析的结果
示例:
  方法: [SWOT分析, 风险矩阵, 专家咨询]
  分析过程: "SWOT显示优势是客户信任，劣势是跨行业经验不足；
            风险矩阵评估为高风险；
            咨询罗汉教授后建议谨慎"
  结论: "理性分析建议拒绝"
```

#### 2.2.8 最终决策 (Final_Decision)

```yaml
描述: 最终的选择和理由
要素:
  - 决策内容: 接受/拒绝/延迟/其他
  - 决策理由: 主要考虑因素
  - 决策依据: 直觉主导/理性主导/平衡
  - 信心度: 0-1

示例:
  决策: 拒绝
  理由: "时机不成熟，风险不可控"
  依据: 直觉主导（直觉不适占70%权重）
  信心度: 0.85
```

#### 2.2.9 结果反馈 (Outcome_Feedback)

```yaml
描述: 决策后的实际结果
时间维度:
  - 短期结果: 1-3个月内
  - 中期结果: 3-12个月内
  - 长期结果: 1年以上

评估:
  - 实际结果: 发生了什么
  - 与预期对比: 是否符合预期
  - 复盘结论: 决策正确/错误/不确定

示例:
  短期结果: 未参与，继续银行工作
  中期结果: 听说项目进展缓慢
  长期结果: 项目6个月后失败，庆幸未参与
  复盘结论: 决策正确
```

#### 2.2.10 模式提炼 (Pattern_Extraction)

```yaml
描述: 从决策中提炼的通用规律
格式: "条件A + 条件B + ... = 决策建议"

类型:
  - 风险识别模式
  - 机会评估模式
  - 决策时机模式
  - 合伙人评估模式

示例:
  模式: "直觉不适 + 时间压力 + 跨行业 = 拒绝"
  条件:
    - 直觉不适（身体信号）
    - 时间压力（<48小时）
    - 跨行业（缺乏经验）
  决策: 拒绝
  置信度: 基于3个案例，准确率100%
```

## 三、标注格式（YAML）

### 3.1 完整标注模板

```yaml
# 案例信息
case_id: CASE-001
case_title: "银行时期核心客户邀请创业"
client_name: "Egbertie"
decision_date: "2018-03"
industry: "金融科技"

# 决策点信息
decision_point:
  dp_id: DP-001-01
  dp_sequence: 1
  total_dps: 3
  
# 核心维度
context:
  description: "银行核心客户（科技公司CEO）邀请参与AI创业项目"
  time_background: "2018年3月"
  location: "深圳"
  stakeholders: ["我", "客户（科技公司CEO）", "银行"]
  trigger: "客户提出邀请"
  decision_type: "是否参与"  # 是/否/选择/退出

information_completeness:
  score: 0.3
  known:
    - "客户信任度高"
    - "行业前景好"
    - "大致团队规模"
  unknown:
    - "详细财务状况"
    - "核心技术细节"
    - "其他合伙人背景"

time_pressure:
  level: "高"  # 高/中/低
  deadline: "48小时内"
  reason: "客户需要快速启动"

interest_conflict:
  my_interests: ["合规性", "职业安全", "长期发展"]
  their_interests: ["快速启动", "资源整合", "银行背景背书"]
  conflicts: ["合规要求 vs 快速启动"]
  coordination_possible: false

cognitive_biases:
  - type: "损失厌恶"
    severity: "高"
    description: "怕失去银行稳定工作"
  - type: "禀赋效应"
    severity: "中"
    description: "高估客户关系价值"
  - type: "现状偏见"
    severity: "高"
    description: "倾向维持现状"

intuition_signal:
  body: "胃部不适，隐隐作痛"
  emotion: "焦虑，不安"
  description: "总觉得哪里不对"
  intensity: "强"

rational_analysis:
  methods: ["SWOT分析", "风险矩阵", "专家咨询"]
  process: "SWOT显示优势是客户信任，劣势是跨行业经验不足；风险矩阵评估为高风险；咨询罗汉教授后建议谨慎"
  conclusion: "理性分析建议拒绝"

final_decision:
  choice: "拒绝"
  reason: "时机不成熟，风险不可控"
  basis: "直觉主导"  # 直觉主导/理性主导/平衡
  confidence: 0.85

outcome_feedback:
  short_term: "未参与，继续银行工作"
  medium_term: "听说项目进展缓慢"
  long_term: "项目6个月后失败"
  vs_expectation: "符合预期"
  conclusion: "决策正确"

pattern_extraction:
  pattern: "直觉不适 + 时间压力 + 跨行业 = 拒绝"
  conditions:
    - "直觉不适（身体信号）"
    - "时间压力（<48小时）"
    - "跨行业（缺乏经验）"
  recommendation: "拒绝"
  evidence_count: 3
  accuracy_rate: 1.0

# 训练标签（用于价值网络）
training_labels:
  decision_quality: 1.0  # 0-1，基于结果
  pattern_type: "风险规避型"
  risk_level: "高"
  urgency_level: "高"
  complexity_level: "中"

# 元数据
metadata:
  annotated_by: "满意妞"
  annotated_at: "2026-03-15"
  verified: false
  notes: "典型案例，直觉与理性一致"
```

## 四、标注流程

### 4.1 标注步骤

```
步骤1: 案例阅读（10分钟）
  └── 完整阅读案例材料
  └── 标记关键决策点
  └── 初步理解情境

步骤2: 信息提取（15分钟）
  └── 提取10个维度的信息
  └── 不确定处标记待确认
  └── 填写基础字段

步骤3: 模式提炼（10分钟）
  └── 分析决策逻辑
  └── 提炼通用模式
  └── 验证模式合理性

步骤4: 标签标注（5分钟）
  └── 标注训练标签
  └── 评估决策质量
  └── 分类模式类型

步骤5: 审核确认（10分钟）
  └── 自检完整性
  └── 交叉验证
  └── 最终确认
```

### 4.2 质量控制

| 检查项 | 标准 | 方法 |
|--------|------|------|
| 完整性 | 所有必填字段已填写 | 自动检查 |
| 一致性 | 各维度间无矛盾 | 逻辑校验 |
| 准确性 | 信息提取正确 | 抽样复核 |
| 可用性 | 可用于训练 | 专家评审 |

### 4.3 审核流程

```
标注员完成 → 自检 → 交叉审核 → 专家抽查 → 入库
    ↓          ↓         ↓          ↓
   50分钟    10分钟    20分钟     10分钟
```

## 五、标注工具

### 5.1 命令行工具

```bash
# 创建新标注
dp-annotation create --case CASE-001 --dp 1

# 编辑标注
dp-annotation edit DP-001-01

# 验证标注
dp-annotation validate DP-001-01

# 导出标注
dp-annotation export --case CASE-001 --format yaml

# 统计进度
dp-annotation stats --by case/pattern/quality
```

### 5.2 批量处理

```bash
# 批量创建
dp-annotation batch-create --cases case-list.txt

# 批量验证
dp-annotation batch-validate --dps dp-list.txt

# 生成报告
dp-annotation report --month 2026-03
```
