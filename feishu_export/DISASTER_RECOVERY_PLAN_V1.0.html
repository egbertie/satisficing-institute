概述

1.1 目的

本文档定义了满意解研究所的灾备重建复刻方案，确保在发生系统故障、数据丢失或其他灾难性事件时，能够快速恢复业务运营，最大限度地减少数据丢失和业务中断。

1.2 适用范围

本方案涵盖满意解研究所的所有核心数字资产，包括： - AI组织运作所需的全部配置文件 - 组织架构与角色定义 - 知识库与研究成果 - 自动化脚本与定时任务 - API密钥与访问凭证

1.3 关键指标

指标 目标值 说明
RTO (恢复时间目标) ≤ 2小时 从灾难发生到系统完全恢复的时间
RPO (恢复点目标) ≤ 24小时 允许的最大数据丢失时间窗口
备份成功率 ≥ 99% 每月备份任务成功执行的比例
恢复测试频率 每月1次 定期进行恢复演练的频率

数据类型 备份频率 保留策略 存储位置

2.2 备份策略详解

2.2.1 每日自动备份 (Daily Auto Backup)

执行时间: 每天 02:00 (Asia/Shanghai)

包含内容: - 所有Markdown文档 (.md) - 记忆文件 (memory/.md, MEMORY.md) - 配置文件 (.env, .config/) - 定时任务配置 (cron/jobs.json) - 脚本文件 (.scripts/)

存储格式:

/backups/daily/YYYY-MM-DD/
├── workspace-md/
├── memory/
├── config/
├── cron/
└── scripts/

2.2.2 每周全量备份 (Weekly Full Backup)

执行时间: 每周日 03:00 (Asia/Shanghai)

包含内容: - 每日备份的所有内容 - Skill文件完整备份 (skills/) - 知识库完整备份 (knowledge_base/) - personas/ 目录 - 完整Git仓库状态

存储格式:

/backups/weekly/YYYY-MM-DD/
├── daily/              # 包含所有日常备份内容
├── skills/
├── knowledge_base/
├── personas/
└── git-snapshot/

2.2.3 变更触发备份 (Change-Triggered Backup)

触发条件: - .env 文件修改 - openclaw.json 修改 - cron/jobs.json 修改 - Skill文件修改

执行方式: 通过 Git Hook 或文件系统监控触发即时备份

2.3 保留策略

备份类型 保留时长 自动清理
每日备份 30天 是

变更触发备份 30天 是

恢复流程

4.1 灾难分级

级别 定义 示例 响应时间 RTO
P0-紧急 完全不可用 系统崩溃、数据全部丢失 15分钟 1小时
P1-严重 核心功能受损 关键配置丢失、API失效 30分钟 2小时
P2-一般 非核心功能受损 部分文档丢失、Skill失效 2小时 4小时
P3-轻微 轻微影响 单个文件丢失、格式错误 24小时 48小时

4.2 P0级灾难恢复流程

步骤1: 灾难确认与通报 (0-15分钟)

  1. 确认灾难范围 bash # 检查系统状态 openclaw status df -h ls -la /root/.openclaw/

  2. 启动应急响应

  3. 通知相关干系人
  4. 创建灾难恢复工单
  5. 冻结所有非紧急操作

步骤2: 恢复准备 (15-30分钟)

  1. 确定恢复点 bash # 查看最新可用备份 ls -lt /backups/daily/ | head -5 ls -lt /backups/weekly/ | head -3

  2. 准备恢复环境

  3. 确保目标环境可用
  4. 验证存储空间充足
  5. 准备恢复脚本

步骤3: 数据恢复 (30-60分钟)

  1. 恢复核心配置 bash # 执行自动恢复脚本 python3 /root/.openclaw/workspace/scripts/backup-manager.py restore --date YYYY-MM-DD --type full

  2. 手动验证关键文件 ```bash # 检查.env cat /root/.openclaw/workspace/.env

# 检查openclaw.json cat /root/.openclaw/openclaw.json

# 检查定时任务 cat /root/.openclaw/cron/jobs.json ```

步骤4: 服务恢复 (60-75分钟)

  1. 重启服务 bash openclaw gateway restart

  2. 验证定时任务 bash openclaw cron list

步骤5: 功能验证 (75-90分钟)

  1. 执行验证清单
  2. [ ] API连接测试
  3. [ ] 定时任务检查
  4. [ ] 文档完整性检查

  5. [ ] Skill功能测试

  6. 生成恢复报告 bash python3 /root/.openclaw/workspace/scripts/backup-manager.py verify --report

4.3 P1级灾难恢复流程

场景: 配置文件丢失

# 1. 定位最近的配置备份
BACKUP_DATE=$(ls -lt /backups/daily/ | grep -E '^d' | head -1 | awk '{print $9}')

# 2. 恢复.env
cp /backups/daily/$BACKUP_DATE/config/.env /root/.openclaw/workspace/.env

# 3. 恢复openclaw.json
cp /backups/daily/$BACKUP_DATE/config/openclaw.json /root/.openclaw/openclaw.json

# 4. 恢复定时任务
cp /backups/daily/$BACKUP_DATE/cron/jobs.json /root/.openclaw/cron/jobs.json

# 5. 验证
openclaw status
openclaw cron list

场景: 部分文档丢失

# 1. 确定丢失的文件
# 2. 从备份恢复特定文件
cp /backups/daily/YYYY-MM-DD/workspace-md/MISSING_FILE.md /root/.openclaw/workspace/

# 3. 验证文档完整性
python3 /root/.openclaw/workspace/scripts/backup-manager.py verify --doc-file MISSING_FILE.md

4.4 单文件恢复流程

# 使用备份管理器恢复单个文件
python3 /root/.openclaw/workspace/scripts/backup-manager.py restore-file \
  --backup-date YYYY-MM-DD \
  --source-path memory/2026-03-10.md \
  --target-path /root/.openclaw/workspace/memory/2026-03-10.md

版本管理机制

5.1 版本号规则

采用语义化版本控制 (Semantic Versioning):

主版本号.次版本号.补丁号
  X    .    Y    .    Z
版本类型 递增条件 示例
主版本号 (X) 重大架构变更、破坏性更新 1.0.0 → 2.0.0
次版本号 (Y) 功能新增、重大优化 1.0.0 → 1.1.0
补丁号 (Z) Bug修复、小调整 1.0.0 → 1.0.1

安全管理重点

7.1 API密钥安全

7.1.1 存储安全

  • ✅ API密钥仅存储于 .env 文件
  • ✅ .env 文件已添加至 .gitignore
  • ✅ 加密备份中包含 .env 的加密版本
  • ✅ 定期轮换API密钥 (建议每90天)

7.1.2 传输安全

  • ✅ 仅通过加密通道传输备份
  • ✅ 云存储使用服务端加密
  • ✅ 本地备份目录设置严格权限 (700)

7.1.3 访问控制

# 设置备份目录权限
chmod 700 /backups
chmod 600 /backups/daily/*
chmod 600 /backups/weekly/*

# 设置.env权限
chmod 600 /root/.openclaw/workspace/.env

7.2 备份加密

7.2.1 加密策略

数据类型 加密方式 密钥管理
API密钥备份 AES-256-GCM 主密钥存储于安全位置
配置文件备份 AES-256-GCM 同上
一般文档备份 可选加密 同上

7.2.2 加密命令

# 加密备份
python3 /root/.openclaw/workspace/scripts/backup-manager.py backup --encrypt

# 解密恢复
python3 /root/.openclaw/workspace/scripts/backup-manager.py restore --date YYYY-MM-DD --decrypt
