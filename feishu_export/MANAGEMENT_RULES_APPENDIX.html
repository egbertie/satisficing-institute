九、外部工具使用策略

9.1 选型原则

对于小型Skill，优先使用外部API/服务，逐步自建替代

阶段 策略 示例
第一阶段 链接外部API快速解决问题 GitHub Actions, Healthchecks.io
第二阶段 运行中积累需求和经验 记录痛点和优化点
第三阶段 自建Skill替代外部依赖 根据实际需求自建
第四阶段 完全自主可控 内部系统闭环

9.2 当前使用的外部工具

工具 用途 替代计划
GitHub Actions CI/CD自动化 第三阶段自建
Healthchecks.io 定时任务监控 第三阶段自建
Uptime Kuma API健康监控 第三阶段自建
Notion API 知识库同步 长期外部依赖
飞书API 消息通知 长期外部依赖

十、定期审查机制

10.1 API与Skill双周审查 [RULE]

频率: 每两周（隔周的周五） 负责人: 满意妞（主控AI） 审查对象: 所有使用的API、Skill、外部工具

审查清单

检查项 检查内容 行动标准
使用频率 过去两周调用次数 <5次考虑删除
稳定性 成功率、错误率 <95%考虑替代
成本 费用变化趋势 增长>50%预警
需求匹配 是否满足当前需求 不匹配考虑优化
替代方案 是否有更好的选择 有则评估切换

审查输出

  1. 审查报告: /root/.openclaw/workspace/docs/BIWEEKLY_REVIEW_YYYY-MM-DD.md
  2. 优化建议清单: 待新建/删除/合并的Skill
  3. 成本分析: API费用趋势和优化建议

审查流程

周四 23:00 - 自动收集数据（调用次数、错误率、费用）
周五 09:00 - 生成审查报告初稿
周五 14:00 - Egbertie审阅并确认优化方案
周五 18:00 - 更新配置和执行优化

10.2 优化决策标准

情况 决策
使用频率低 + 需求仍在 保留但降低优先级
使用频率低 + 需求消失 标记弃用，下月删除
稳定性差 + 有替代 切换至替代方案
稳定性差 + 无替代 自建Skill解决
成本增长 + 有免费替代 迁移至免费方案
需求变化 + 功能过剩 降级至轻量方案
